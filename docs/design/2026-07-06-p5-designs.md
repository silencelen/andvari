## 1. LOW-3 — real client IP behind loopback front-ends

**Decision: do NOT install ktor's `ForwardedHeaders`/`XForwardedHeaders` plugin** (candidate site App.kt:118-121). The plugin rewrites `request.origin` globally and unconditionally, which would (a) let a non-loopback LAN peer spoof XFF and (b) perturb the `/metrics` loopback gate's semantics. Instead, keep `origin` raw and make `clientIp()` itself proxy-aware.

**Tailscale-serve header determination:** the repo has no local knowledge doc naming the header (grepped `ops/`, `docs/`, `spec/` — only the audit's own text and the spec 03 §8 `CF-Connecting-IP` mention). Public Tailscale docs say `tailscale serve` sets `X-Forwarded-For` (plus `Tailscale-User-*` identity headers), but per the task's uncertainty rule the trusted-header list is **env-config-gated with sane defaults**: `CF-Connecting-IP` first (cloudflared path; set authoritatively by the CF edge), then `X-Forwarded-For` (tailscale-serve path). Verifying the actual header on CT 122 at deploy time is an open question; the env override absorbs any surprise without a code change.

### Files

- **`server/src/main/kotlin/io/silencelen/andvari/server/Auth.kt:27`** — replace
  `fun ApplicationCall.clientIp(): String = request.origin.remoteHost` with:
  - `fun ApplicationCall.clientIp(config: Config): String` — computes `peer = request.origin.remoteAddress`, `peerIsLoopback` via the same `InetAddress.getByName(peer).isLoopbackAddress` idiom already used at App.kt:175-176, then delegates to the pure helper with `request.origin.remoteHost` as fallback.
  - `internal fun pickClientIp(peerIsLoopback: Boolean, header: (String) -> String?, trustedHeaders: List<String>, fallback: String): String` — pure, unit-testable without a call: if `!peerIsLoopback` return `fallback` (raw XFF from a non-loopback peer is NEVER trusted); else for each trusted header name in order: take the header; for `X-Forwarded-For` take the **rightmost** comma-separated entry (the one appended by the trusted loopback proxy — never walk deeper into the client-forgeable chain); trim; accept only if it is an **IP literal** (guard: `InetAddress.getByName` does DNS on hostnames, so gate with an IPv4/IPv6 literal check, e.g. regex or `sun`-free manual parse) and non-loopback-string; else fall through. Final fallback: `fallback`.
- **`server/src/main/kotlin/io/silencelen/andvari/server/App.kt`** — all 11 `call.clientIp()` sites become `call.clientIp(config)`: :223 (prelogin rate key), :231 (register), :236 (login rate key), :239 (login), :245 (refresh), :260 (password change), :275/:280 (totp confirm/disable), :325 (push), :339 (escrow self), :404 (policy — see item 7, this call moves into `setPolicy`). Rate-bucket keys `prelogin:<ip>` / `login:<ip>` (App.kt:223/236) automatically become per-client again.
- **`server/.../App.kt:172-182` `/metrics`** — UNTOUCHED (it reads `request.origin.remoteAddress` directly and must keep never trusting XFF; the comment at :173-174 stays).
- **`server/src/main/kotlin/io/silencelen/andvari/server/Config.kt`** — add `val trustedIpHeaders: List<String> = listOf("CF-Connecting-IP", "X-Forwarded-For")` (constructor, after :26); in `fromEnv` (:42-57) parse `ANDVARI_TRUSTED_IP_HEADERS` (comma-separated, blank entries dropped; unset → default).
- **`ops/andvari.env.example`** — new commented block after the enum-secret block (:23):
  `# Forwarded-IP headers trusted ONLY when the direct peer is loopback (tailscale serve /`
  `# cloudflared both terminate on 127.0.0.1). Raw XFF from non-loopback peers is always ignored.`
  `# ANDVARI_TRUSTED_IP_HEADERS=CF-Connecting-IP,X-Forwarded-For`
- **`spec/03-wire-protocol.md` §8 (:158-160)** — extend the rate-limit bullet: client IP for rate keys and audit rows is derived from `CF-Connecting-IP`, then rightmost `X-Forwarded-For`, **only when the direct TCP peer is loopback**; otherwise the socket peer address is used and forwarded headers are ignored.

### Tests (extend `server/src/test/.../AuditHardeningTest.kt`)

`testApplication`'s origin resolves as a loopback host, so the gate opens in-test:
1. Login with header `CF-Connecting-IP: 203.0.113.9` → `auditRows(admin,"login")` last row `ip == "203.0.113.9"`.
2. Login with `X-Forwarded-For: 6.6.6.6, 100.101.102.103` (no CF header) → ip == `100.101.102.103` (rightmost wins; forged left entry ignored).
3. Header priority: both headers present → CF value wins.
4. Garbage header (`X-Forwarded-For: not-an-ip`) → falls back to the raw peer (no crash, no garbage in audit).
5. Pure-function tests for `pickClientIp`: `peerIsLoopback=false` ignores all headers; empty/blank header falls through; IP-literal guard rejects hostnames.

## 3. LOW-9 — WS auth via single-use ticket; drop token-in-URL

### Spec delta FIRST

## Adversarial verification — zk_holds=true, verdict=issues

- **[major]** LOW-9 web change adds a ticket-mint round-trip BEFORE the WS opens, but the design explicitly forbids editing live.e2e.test.ts, which uses a fixed 300 ms settle window before client A pushes. Notifier.notifyRev is fire-and-forget to currently-registered sockets — no buffering/replay. With the old code events() opened the WS synchronously and 300 ms sufficed for connect+register; the new async events() must complete POST /events/ticket AND the WS upgrade+register within that same 300 ms or A's push notifies zero listeners and the bell is lost permanently, failing `expect(bellRev).toBeGreaterThan(0)`. On a gradle-flock-serialized 8-12 GB CI host running vitest+Netty concurrently, this is a real flake/red on the exact gate the design names as its green/red proof for this item.
  - **Fix:** Don't rely on an unchanged 300 ms window. Have events() expose an awaitable 'socket-open' signal (e.g. return {close, ready:Promise} or an onOpen callback) and await it in the e2e before A pushes; OR widen the settle window in live.e2e.test.ts (accept that this item DOES touch the e2e test) and poll for registration. Update the design's 'no edits to live.e2e.test.ts' claim accordingly.
- **[minor]** Netty requestReadTimeoutSeconds vs idle WebSocket: the design assumes the read timeout does NOT reap upgraded WS connections because Ktor pings every 30 s and browser auto-pongs create inbound traffic (default 300 s > 60 s). Whether Netty's requestReadTimeoutSeconds resets on post-upgrade WS frame reads is genuinely uncertain (flagged only as manual CT 122 verify). If it does apply to the upgraded connection, every healthy idle dirty-bell drops at the timeout interval and web sync silently stalls until remount (web has no poll fallback — v1 cut).
  - **Fix:** Verify on CT 122 BEFORE enabling in production; until confirmed, default ANDVARI_REQUEST_READ_TIMEOUT_S to 0 (off) rather than 300, or explicitly exempt the WS route. Keep the ping keepalive regardless. Do not ship the 300 s default as on-by-default until the WS-survival test in the plan passes.
- **[minor]** Pre-deploy open web tabs: the design deletes the server's `?access=` WS path and claims 'web deploys with the server — no compat window.' But a browser tab already running the OLD bundle (loaded before deploy) still calls events()?access=token; the upgraded server closes it VIOLATED_POLICY. Web has no 5-min poll fallback (v1 cut), so that tab's dirty-bell dies silently until a manual reload.
  - **Fix:** Accept and document as a known transient (single household, self-heals on reload), OR keep accepting `?access=` for a short grace window (it is no worse than today's posture) and remove it in a follow-up once no old tabs remain. Add a note to the deploy runbook to hard-reload open web tabs.
- **[minor]** INFO-1 correlation rationale is internally inconsistent: the design sets register audit meta to null AND sets invite_create meta to tokenHash.take(12) 'to keep create↔redeem correlation.' But register logs neither the email nor the tokenHash, so there is nothing on the redeem side to join the tokenHash prefix to — the advertised correlation is unachievable.
  - **Fix:** If create↔redeem correlation is wanted, also record the tokenHash prefix (not the email) in the register audit meta; otherwise drop invite_create meta to null too and state honestly that the correlation is intentionally not retained.
- **[minor]** INFO-2 short-form entry displays the full grouped fingerprint on-screen immediately above the 'type the first 16 characters from your printed recovery sheet' input, so a user types what is already rendered rather than performing the intended out-of-band comparison against the printed sheet. Combined with the design's own concession that a malicious server already dominates the web client (T6/R2), this is spec-04 §2(3) conformance/theater, not a real strengthening.
  - **Fix:** When requiring short-form entry, do NOT also display the fingerprint the user is supposed to transcribe (or blur it until after entry), so the typed value must come from the sheet. Keep the honest note that native clients remain the trust anchor for high-value escrow confirmation.
- **[minor]** LOW-6 semaphore/accounting footguns: the per-user InFlight entries in the ConcurrentHashMap are never evicted (unbounded-in-principle, negligible per household but worth noting), and sem.release()/bytes-subtract in finally must only run when tryAcquire actually succeeded — the RateLimited throw on failed acquire must occur BEFORE entering the try/finally that releases, or a non-acquired path will over-release the semaphore and corrupt the concurrency cap.
  - **Fix:** Acquire outside the try; enter the release-finally only after a successful acquire; track this upload's added bytes in a local var and subtract exactly that in finally. Optionally evict empty InFlight entries opportunistically. Add a unit test that a failed tryAcquire does not release a permit.

# Design: csv-import (Chrome/Edge + Firefox CSV → personal vault)

100% client-side per spec/06: read file bytes on-device → RFC-4180 parse → build `ItemDoc`s → encrypt → normal `/sync/push` in chunks. The file never uploads; the report is shown, never stored. **Zero server changes, zero wire changes, zero schema changes** — deployed Android 0.2.4 / desktop 0.2.0 / CT 122 are untouched except that new clients push ordinary put mutations.

Order of work (locked): spec → vector-gen → Kotlin core → web TS → UIs.

## 1. Spec deltas (land FIRST)

### `spec/06-import-formats.md` (41 lines today; §1 L7, §2 L18, §3 L28)
Extend — the current text states intent but under-specifies parsing, so both impls could legally diverge. Add:

- **New §4 "Normative parsing algorithm"** (insert after §3, ~L41):
  - Limits, in check order: raw byte length > 10 MiB → reject `too_large` (before decode); data rows (excl. header) > 10 000 → reject `too_many_rows`.
  - Decode: UTF-8, lenient (invalid sequences → U+FFFD, matching Kotlin `decodeToString()` and TS `TextDecoder` defaults). Strip one leading BOM (EF BB BF / U+FEFF).
  - RFC-4180 state machine: records end at CRLF or bare LF; one trailing newline ignored; `"` opens a quoted field only at field start; `""` inside quotes → literal quote; quoted fields may contain commas/CR/LF; **lenient rule**: `"` inside an *unquoted* field is a literal character; content after a closing quote before the next delimiter → per-row error `bad_quote` (row skipped, import continues).
  - Header detection (row 1, names trimmed + lowercased): contains `name,url,username,password` → Chrome/Edge (`note` **or** `notes` column optional — Chrome emits `note`); contains `url,username,password` plus any of `guid|httprealm|formactionorigin` → Firefox; otherwise reject `unrecognized_header`. Columns mapped **by header name**, not position (handles old 4-col Chrome).
  - Row errors: field count ≠ header count → `wrong_field_count`; reported with the **1-based physical line number of the row's first character** (embedded newlines make row index ≠ line number — pinned).
  - Skip rule (both formats): empty username AND empty password → counted `skippedEmpty`.
  - **Name fallback host extraction** (pinned algorithm so impls match; no URL library): trim → strip `scheme://` (`[A-Za-z][A-Za-z0-9+.-]*://`) → cut at first `/ ? #` → drop userinfo up to last `@` → if starts `[`, host = bracket contents (IPv6) else strip trailing `:digits` → lowercase. Empty result → trimmed raw url → else `"Imported login"`. Chrome: used only when `name` empty; Firefox: always.
  - **Dedupe/rename** (file-internal, in row order after skips/errors): exact `(url, username, password)` repeat → drop, `collapsed++`; same `(url, username)` with a different password → import with name `"<name> (n)"` where n = 2-based index of distinct passwords within the group, and record the final name in `flagged`.
  - Report fields: `imported`, `skippedEmpty`, `collapsed`, `flagged: [names]`, `errors: [{line, code}]`. Firefox `timePasswordChanged` is parsed (vector-covered) but **not** materialized into `passwordHistory` in v1 (no retired password exists to store) — state this explicitly.
- **New §5 "Test vectors"**: document `test-vectors/import.json` shape (below) and that the reject cases are recipe-constructed to keep the file small.

```kotlin
/** Bulk import into the personal vault: chunked pushes (server caps batches at 200,
 *  Service.kt), one pull at the end. Queue durability makes a crash resumable. */
suspend fun importAll(docs: List<ItemDoc>, chunkSize: Int = 100,
                      onProgress: ((done: Int, total: Int) -> Unit)? = null) {
    val vaultId = account.personalVaultId
    docs.chunked(chunkSize).fold(0) { done, chunk ->
        push(chunk.map { putMutation(account.newItemId(), vaultId, it, 0) })
        (done + chunk.size).also { onProgress?.invoke(it, docs.size) }
    }
    pull()
}
```

## ZK statement

Import never changes what the server can see. The CSV bytes are read and parsed entirely on the client (core commonMain / web src/import — pure functions, no network); the resulting ItemDocs are encrypted client-side by the existing Account.encryptItem path (XChaCha20 envelope under the personal-vault VK, AD-bound to userId/vaultId/itemId per spec 02 §2) and delivered as ordinary put mutations on the existing /sync/push endpoint. The server therefore receives exactly what the spec 02 §5 items row already permits — itemId/vaultId/rev/timestamps/flags/formatVersion/attachmentIds/ciphertext blob + ciphertext size — and nothing else: no new endpoints, no new wire fields, no schema change, no server source touched, the file itself never uploads, and the import report lives only in client memory (shown, not stored). Keys are never persisted in plaintext (the plan/report hold row plaintext only in RAM, same as the existing editor). The only server-observable side effect is a burst of puts (count/timing/sizes), which is traffic analysis the ZK contract already accepts under spec 02 §5 / spec 05.

- Chrome emits header column `note`; do we also accept `notes` (hand-edited files)? Design pins accepting both, case-insensitive — confirm before it lands in spec §4.
- Android OpenDocument MIME filter: providers variously tag CSVs text/csv, text/comma-separated-values, application/vnd.ms-excel or octet-stream — design uses `*/*` (matching the attachment picker); acceptable, or restrict and risk greyed-out files?
- Unrecognized-header handling: whole-file reject (design) vs best-effort positional parse — reject chosen for safety (wrong column order would silently swap username/password); confirm.
- Is there any appetite for a server-side per-user item-count quota before allowing 10k-item imports (none exists today; ~10k ciphertext rows ≈ low MBs on CT 122)? Design adds none.

## Adversarial verification — zk_holds=true, verdict=issues

- **[blocker]** Queue-overflow wedge + retry duplication in the Kotlin importAll path. SyncEngine.flushQueue (SyncEngine.kt:139-155) sends the ENTIRE pending queue as ONE PushRequest, and the server hard-rejects any batch >200 with BadRequest("batch_too_large") for the WHOLE request (Service.kt:295) — nothing is applied, nothing is dequeued. importAll makes pending>200 reachable for the first time: import a 150+-row CSV while offline/flaky → chunk 1 (100 mutations) is enqueued, api.push throws, importAll aborts; user taps Import again (each attempt mints FRESH itemIds and mutationIds) → pending=200; third attempt → pending=300 → every subsequent flushQueue/sync() gets 400 batch_too_large, throws, dequeues nothing, and pull() never runs — sync is permanently wedged for the session (until app restart clears the in-memory queue). Even the pending=200 case is bad: when the network returns, all 200 apply and every imported item is duplicated even though the user never saw a successful import. The design's claim that '/sync/push has no rate limiter' and 'chunk 100 stays under the 200 cap' is true per-chunk but false for the shared queue that push() drains.
  - **Fix:** Two changes before implementation: (1) make flushQueue chunk cache.pending() into <=200-mutation PushRequests (fixes the wedge for all callers, not just import); (2) make import retries idempotent by generating itemIds AND mutationIds once at plan time (store them in ImportPlan, e.g. docs: List<Pair<String, ItemDoc>> or a per-plan importId + deterministic per-row ids) and reusing them on every attempt — the server's (deviceId, mutationId) idempotency table (Service.kt:311-314) then replays results instead of duplicating items. Optionally also fail fast: probe with a cheap sync() before enqueueing anything, and disable the Import button while cache.pending() is non-empty.
- **[major]** The design's crash-durability premise is false in the field: 'Queue durability makes a crash resumable' (importAll doc comment) and 'enqueue-then-flush -> crash-durable' rest on a persistent VaultCache, but BOTH shipped clients — and the design's own Android/desktop UIs — wire InMemoryVaultCache (AndvariViewModel.kt:250, DesktopState.kt:221; VaultCache.kt:38 'no persistence across process death'). A crash/kill mid-import (10k rows = up to 100 network round-trips, plenty of window on Android) loses the queue AND the plan: chunks already pushed are on the server, the rest are gone, and re-running the import duplicates every already-applied item because v1 has no dedupe against existing vault items (cut #3). The user's only remediation is manually deleting up to thousands of duplicates.
  - **Fix:** Same plan-time-stable itemId/mutationId fix as the blocker makes re-running the SAME plan server-idempotent on the same device — the mutations table replays 'applied' for chunks that landed and only the missing rows insert. Then change the UI contract: after a failed/interrupted import, keep the ImportPlan in memory and offer 'Resume import' (re-run the identical plan) rather than making the user re-pick the file (a re-parse mints a new plan; if you derive ids deterministically from a plan-level importId + row index, persist/redisplay that importId so even a re-parse can resume). Also correct the doc comment — with InMemoryVaultCache the queue is retry-durable within a process, not crash-durable.
- **[major]** Web importDocs has no failure/resume semantics at all and is worse than the Kotlin path: web VaultStore.push (store.ts:183-189) has NO queue — it POSTs directly — so a network error at chunk k of a 10k import throws out of importDocs with k*100 items committed server-side and no record of progress. The design's ImportPanel flow (Confirm -> store.importDocs -> report) doesn't say what the panel shows on a mid-import throw; the natural implementation (catch -> error message -> user re-clicks Confirm) re-runs planImport docs with fresh crypto.randomUUID() mutationIds/itemIds (store.ts:120-122) and duplicates all k*100 already-imported items. This is the most likely real-world failure mode for the primary import surface (web, per the manual smoke plan).
  - **Fix:** Mirror the Kotlin fix in the TS twin: planImport assigns itemIds and mutationIds once into the plan; importDocs tracks a resume index (or simply re-sends everything and relies on server (deviceId,mutationId) idempotent replay); ImportPanel's error state keeps the SAME plan object and offers Retry against it, only discarding the plan after the final report renders. Specify this in the design's section 5 so both impls and the vectors/tests pin it (add a server integration test that kills the connection mid-import and re-runs importAll with the same plan, asserting no duplicates).
- **[minor]** Spec section 4 state machine leaves two divergence points unpinned: (1) EOF while inside a quoted field (unterminated quote) — one impl may emit the partial field as a row, the other may error; (2) a lone CR not followed by LF — record terminator, literal character, or error? Both are trivially reachable from hand-edited CSVs, and since vector expectations are emitted by the Kotlin reference impl, the TS twin would only be caught if a vector happens to cover it — otherwise verify.sh stays green while the impls disagree.
  - **Fix:** Pin both in spec section 4: EOF inside quotes -> per-row error `bad_quote` at the row's opening line (row skipped, prior rows still import); lone CR outside quotes -> treated as a record terminator equivalent to LF (or literal — either, but pick one). Add one vector file covering unterminated-quote-at-EOF and a lone-CR line ending so import.json enforces it on both implementations.
- **[minor]** Android file-read OOM window: the design copies the attachment-picker pattern (MainActivity.kt:387-412) which does contentResolver.openInputStream(uri).readBytes() — the WHOLE file is read into a ByteArray BEFORE any size check, and the 'size-check before read' via displayName/OpenableColumns SIZE is unreliable (providers may return null size, and content:// streams from cloud providers report no length). A mispicked multi-GB file (the `*/*` MIME filter makes mispicks easy) OOM-kills the app before the 10 MiB `too_large` rejection ever runs.
  - **Fix:** Bound the read itself: read at most MAX_BYTES+1 bytes from the stream (e.g. a loop with a 10 MiB+1 cap) and reject `too_large` the moment the cap is exceeded, instead of readBytes(). Same applies to the desktop path (File.readBytes() after a File.length() pre-check is fine there since length() is reliable for local files — keep the pre-check).
- **[minor]** Post-import burst behavior on live/old clients is unexamined: server pull is unpaged (Repo.pull returns every item since the cursor in one JSON response), so after a 10k import, the fielded Android 0.2.4 / desktop 0.2.0 clients pull ~10k WireItems (several MB) in a single response and decrypt them all at unlock — works but is an untested memory/latency spike on phones; and each of the ~100 chunk pushes fires onChange -> WS notify, so another live device performs up to 100 notify-triggered sync round-trips during the import (harmless per notify-only contract, but chatty). Also every import writes one idempotency row per mutation into the unpruned `mutations` table (10k rows/import/device).
  - **Fix:** No design change strictly required (all flows keep working), but state it: add a note that pull is unpaged and 10k items is the accepted ceiling, and consider coalescing — e.g. importAll could suppress per-chunk onChange fan-out by design acceptance, or simply document the notify burst. Cheap win: raise chunkSize to 200 (the actual server cap) to halve requests, notifies, and idempotency rows — the 100 figure buys nothing once flushQueue chunks correctly.

# Design: durable offline cache for native clients

## 0. Verified current state and the two load-bearing findings

**Finding 1 — grants/vaults/removedGrants are DELTAS, so a durable cursor forces durable grant state.**
`Repo.pull` (`server/src/main/kotlin/io/silencelen/andvari/server/Repo.kt:181-215`) filters ALL four surfaces by `rev > since`: vaults (:183-187), grants (:193-195), items (:202-206), removedGrants (:207-210). `Service.pull` (`Service.kt:289-291`) 410s only when `since < oldestRetainedRev`. So a client that cold-starts with a persisted cursor > 0 will **never re-receive** the grants/vault rows it already consumed. Meanwhile `Account.unlock` (`core/src/commonMain/.../client/Account.kt:126-137`) returns an account with **empty `vaultKeys` and empty `personalVaultId`** — today's code only works because every relaunch pulls from 0 and `SyncEngine.pull` re-applies grants (:43-47) and personal-vault detection (:48-50). **Conclusion: the cache MUST persist, per account: grant rows (vaultId, role, wrappedVk, rev), vault rows (vaultId, type, rev, metaBlob, createdAt), and item ciphertext envelopes — all of which are already server-visible ciphertext/metadata (spec 02 §5 rows `grants`, `vaults`, `items`), so persisting them client-side adds zero plaintext surface.**

**Finding 2 — the VaultCache interface cannot persist ciphertext as-shaped.** `VaultCache.upsertItem(VaultItem)` (`core/.../client/VaultCache.kt:25`) receives only the decrypted `ItemDoc`; the wire envelope (`WireItem.blob`) is dropped in `SyncEngine.pull` (:59-63). Storing plaintext is forbidden; re-encrypting client-side is wrong (keys must not enter the cache layer, and envelopes are non-deterministic). The interface must carry the `WireItem` alongside the decrypted view. This is an in-repo client interface — Android 0.2.4 / desktop 0.2.0 bundle their own `:core`, so changing it breaks no deployed binary and no wire byte.

Also noted: `SyncEngine.pull` currently ignores `resp.removedGrants` entirely, and swallows per-item decrypt failures (`runCatching` :59-63) while still advancing the cursor — an undecryptable item (e.g. a future `formatVersion`) is silently lost until it changes again. The durable cache fixes the second for free by persisting the envelope even when decryption fails (retried on every hydrate, i.e. after an app upgrade).

## 1. Storage choice: hand-rolled SQLite pair, not the SQLDelight plugin (justified)

The brief allows either. After reading the build, I choose a **hand-rolled two-driver SQLite layer**:

- **Zero new version axes.** `gradle/libs.versions.toml` pins Kotlin 2.1.0 / AGP 8.7.3 / Gradle 8.10.2 (:1-3) and already carries `sqlite-jdbc = 3.47.1.0` (:11, :44) — the exact JDBC driver the server runs on the same JVM 17 toolchain. The SQLDelight gradle plugin would add a codegen plugin whose version must co-track Kotlin/AGP (2.0.2 predates Kotlin 2.1; 2.1.x is a new unpinned axis) on flock-serialized 8–12 GB build hosts (`scripts/verify.sh` header). Android side uses the **framework** `android.database.sqlite` — no dependency at all (minSdk 29, `core/build.gradle.kts:61`).
- **The schema doesn't earn a DSL.** Five tables of key/row CRUD, no joins, no reactive queries, no typed projections. SQLDelight's value (compile-checked SQL, Flow queries, migration DSL) is ~unused; its cost (plugin, generated sources, two driver artifacts) is not.
- **The repo already has the pattern.** `core/src/jvmShared/kotlin` compiles one file into both JVM-family targets (lazysodium adapter, `core/build.gradle.kts:26-53`). The cache impl lives there once; only a ~50-line driver shim differs per platform. This matches the house style (hand-rolled HKDF, hand-rolled JDBC helpers in `Repo.kt:27-57`, no ORM anywhere).
- Trade-off accepted: no compile-time SQL checking, hand-written `PRAGMA user_version` migrations. Mitigated by the cache-contract test suite (below) running the REAL durable impl in `verify.sh`.

Spec 05 T3 currently says "SQLDelight stores envelopes" — edited spec-FIRST to say "SQLite" (see §5). Feature keyname stays `sqldelight-offline-cache` for tracking; the doc comment at `VaultCache.kt:14-20` is updated to match reality.

## ZK statement

The server sees zero change: no new endpoint, header, JSON field, or server-side schema row — its visible surface remains exactly the spec 02 §5 table. Everything the new cache writes to disk is a strict subset of what that table says the server already stores: item rows with their AEAD `blob` ciphertext, grant rows with `wrappedVk` ciphertext (VK sealed under the user's UVK per spec 01 §6), vault rows with `metaBlob` ciphertext, the numeric sync cursor, queued mutations whose payload is the same `ItemUpload` ciphertext the wire carries, and (for offline unlock) the `accountKeys` payload the server hands any authenticated device — wrappedUvk/encryptedIdentitySeed ciphertexts plus public salts/params. UVK, MK, VKs, the identity seed, master password, and decrypted ItemDocs never touch the DB: decryption still happens only in memory inside Account/SyncEngine after the user supplies the master password, and lock/relaunch drops all key material. Because client-at-rest ⊆ server-at-rest, a stolen device DB grants an attacker no more than a full server compromise already would (spec 05 T1/T3), and all vault crypto remains client-side.

## Adversarial verification — zk_holds=true, verdict=issues

- **[major]** Grant/vault persistence is specified INSIDE the existing guarded branch. Design §3.6 says 'after account.addPersonalGrant(grant) also cache.upsertGrant(grant)', but addPersonalGrant runs inside `if (grant.role.isNotEmpty() && !account.hasVault(...)) runCatching {...}` (SyncEngine.kt:43-47). Failing scenario 1: ENROLL session — Account.enroll returns an account that already hasVault(personalVaultId) (Account.kt:121), so the first full pull skips the branch, the grant row is never persisted, cursor advances past its rev, and the flagship cold-start relaunch has no VK and no personalVaultId (save() throws 'no key for vault'). The design's own coldStartWithCursor test uses enroll+save and would fail. Failing scenario 2: a grant whose addPersonalGrant throws (future sealed shared-vault grant — runCatching swallows it) is never persisted and never re-sent. Same ambiguity for §3.6 ':48-50 add cache.upsertVault(vault)' relative to the `type=="personal"` if.
  - **Fix:** Specify persistence UNCONDITIONALLY: `cache.upsertGrant(grant)` for every element of resp.grants and `cache.upsertVault(vault)` for every element of resp.vaults, before/independent of the in-memory guard (they are ciphertext rows, same persist-even-if-undecodable rule the design already applies to item envelopes); hydrate()'s runCatching handles undecodable grants on retry.
- **[major]** Durable cursor + crash window silently loses conflict copies forever. In pull(), cache.setCursor(resp.rev) (SyncEngine.kt:65) runs BEFORE materializeConflict (:67). Today a crash in that window is harmless because relaunch re-pulls from 0 and sees conflict=true again. With a durable cursor, a crash (or the existing decrypt-failure path, now upsertItem(wire,null)) after setCursor means the conflict-flagged item never re-sends, hydrate() never re-materializes, and on a single-device account the displaced version stays buried in item_versions with conflict=true forever — violating the 'client materializes conflict copies' locked decision / no-silent-loss goal.
  - **Fix:** Derive materialization from persisted state instead of the transient in-loop list: the schema already stores the conflict column, so after each pull (and after hydrate, post-first-successful-pull to avoid racing another device's flag-clear), scan cached envelopes with conflict=1 && decryptable and materialize them; the flag-clearing push makes it converge and later syncs skip.
- **[major]** 410 resync clears the durable cache BEFORE the replacement full pull succeeds. pull()'s 410 handler is `cache.clear(); api.sync(0)` (SyncEngine.kt:37-40, kept unchanged by the design). With the durable impl, clear() deletes all items/grants/vaults rows on disk; if the follow-up sync(0) fails (network drop, server restart mid-resync) or the process dies between the two calls, the device is left with an EMPTY durable cache — the next offline unlock shows an empty vault. That's a durability regression introduced by this feature (in-memory clear had no cross-launch consequence) and contradicts its own crash-durability pitch.
  - **Fix:** Make resync transactional: fetch the full response first, then inside one SQLite tx wipe items/grants/vaults and apply the full snapshot + new cursor (SqliteVaultCache.tx makes this trivial); only commit on success, keep the queue as designed.
- **[major]** Rev-regression guard is placed AFTER the response has been applied. Design §3.6 puts the guard 'before :65' — i.e., after the grants/vaults/items loops have already mutated cache+memory. Against an honest rolled-back server the response is empty so nothing happens, but against the T1 adversary the guard is meant to mitigate (server replaying old but AD-valid envelopes — rev is deliberately not in AD, spec 02 §2 — e.g. resurrecting a retired password, alongside a low resp.rev), the stale rows are applied and persisted before the throw. This contradicts spec 05 T1 'clients warn, never delete local newer state' and the design's own revRegressionGuard test assertion 'cached items and cursor untouched'.
  - **Fix:** Move the guard to the top of response processing, immediately after the 410 handling and before any grant/vault/item application: `if (!resp.full && resp.rev < cache.cursor()) throw ...`. Optionally also reject full=true responses when since>0 was requested (an honest server only sends full for since=0), closing the trivial malicious-server bypass.
- **[major]** Cached accountKeys silently changes the spec 05 T3 posture and the design does not update the spec for it. Today an unlock REQUIRES an online GET /account/keys with a live session (AndvariViewModel.kt:126-134), so remote device revocation or a password change (which rotates wrappedUvk/kdfSalt and revokes other sessions, spec 03 §3) effectively blocks unlock on a lost device even for someone who knows the master password. With accountKeysJson persisted in Session/DesktopSession, a revoked or password-rotated device unlocks the full cached vault offline with the OLD master password indefinitely — T3's 'remote device revocation kills sync; local data remains ciphertext without ... master password' and the value of post-compromise password change are both weakened, yet the design's only T3 edit is the SQLDelight wording. Also a UX trap: after changing the password on another device, the NEW password fails offline (old kdfSalt/wrappedUvk) while the old one works.
  - **Fix:** (a) Update spec 05 T3 (and spec 02 §8) to state explicitly that offline unlock accepts the master password that was current at last online contact, and that revocation only takes effect at next connectivity; (b) on any definitive server rejection (device-revoked error, 401 invalid session after refresh, WS 'revoked' frame) wipe cached accountKeys AND the cache DB — this also resolves open question 5; (c) alternatively take the design's own open-question escape hatch and cut offline unlock from v1, which removes the whole class.
- **[minor]** signOut deletes the cache DB without first closing the engine. On Windows (the actual desktop target, MSI), sqlite-jdbc holds the file handle and File.delete() silently returns false — the ciphertext DB survives sign-out, violating the new spec 02 §8 MUST ('cache is deleted on sign-out'). Android/Linux happen to work (unlink-while-open).
  - **Fix:** In both clients' signOut(): engine?.close() (which closes the SqlBox) before deleting vault-<userId>.db/-wal/-shm, and assert deletion in the desktop path; also make bind() defensively close any previously bound engine so two connections never share one file.
- **[minor]** dropVault (removedGrants) purges disk rows but the design leaves the revoked vault's decrypted items in the SqliteVaultCache in-memory map and its VK in Account.vaultKeys until lock/relaunch — the user keeps browsing (and can attempt writes to) a vault they lost, and Account has no removeVault API to call.
  - **Fix:** Have dropVault also evict the vault's items from the in-memory working map (same class, trivial), and add Account.removeVault(vaultId) invoked from the removedGrants loop so pushes fail fast client-side.
- **[minor]** The Android SqlBox actual is uncovered by any automated test and has two known framework traps the design doesn't mention: (1) rawQuery selectionArgs are String[] only — binding a Long through the Any? shim degrades to TEXT and SQLite comparisons of INTEGER columns against TEXT binds silently match nothing (fine for today's PK-by-TEXT queries, a landmine for any future rev-filtered query); (2) execSQL of row-returning PRAGMAs (journal_mode=WAL) throws — must use enableWriteAheadLogging()/rawQuery instead.
  - **Fix:** Document both constraints in the SqlBox contract (or restrict query args to String in the interface and convert explicitly), use SQLiteDatabase.enableWriteAheadLogging() in the Android actual, and add a contract-test case that filters on a numeric column so the JDBC impl at least pins the intended semantics.
- **[minor]** offlineCacheAllowed enforcement has an offline hole and the backup question interacts with tokens: bind() checks `_ui.value.policy?.offlineCacheAllowed != false`, but policy is a best-effort unauthenticated fetch at start (AndvariViewModel.kt:66) — when it fails, policy==null is treated as allowed, so a policy-forbidden device that cold-starts offline still creates the durable DB. Separately, Android auto-backup will now upload vault-*.db AND SharedPreferences containing refreshToken+accountKeysJson to Google.
  - **Fix:** Re-evaluate the cache decision on every successful policy fetch (delete the DB when a later fetch says false, not only at bind), and answer open question 4 affirmatively: add a fullBackupContent/dataExtractionRules exclusion for vault-*.db and the andvari prefs file (the token exposure is pre-existing but this feature doubles what's in the backup).

## 0. Two latent defects this feature trips over (fix first, they gate correctness)

1. **Save always targets the personal vault, even for existing items.** `core/.../client/SyncEngine.kt:104` (`val vaultId = account.personalVaultId`) and `web/src/vault/store.ts:150` (`const vaultId = this.account.personalVaultId`) ignore the existing item's vault. The moment a client edits an item that lives in a shared vault, it re-encrypts under the personal VK with AD `andvari/v1|item|{personalVaultId}|…` but the server row keeps the shared `vaultId` (the UPDATE at `server/.../Service.kt:355-358` never touches `vaultId`) → the blob's AD no longer matches its row → **every other member gets a permanent decrypt failure**. 2. **New-grant backfill is missing.** `server/.../Repo.kt:183-206`: `pull()` filters vaults by `v.rev>?` and items by `i.rev>?`. A member added at rev 5000 whose cursor is 5000 receives the *grant* (its rev is new) but **never the vault row or any pre-existing items** (their revs are old). Grants sync "exists" but is unusable for sharing. Fix: deliver vault + items when *either* the object or the caller's grant on that vault is newer than `since` (§3.2).

## ZK statement

The server continues to see exactly the spec 02 §5 plaintext table and nothing new. Every cryptographic operation stays client-side: the shared VK is generated by the creator, wrapped for the creator under their UVK (an AEAD envelope, opaque to the server) and for each member as crypto_box_seal to the member's identityPub (opaque ciphertext with the vaultId bound inside the sealed payload, verified by the recipient — the server cannot forge, redirect, or read it). The new grants.sealedVk column stores only this ciphertext, and the §5 grants row already declared "wrapped/sealed VK ciphertext". Vault names remain metaBlob ciphertext under VK; item blobs remain AD-bound to (vaultId,itemId) so the server still cannot swap ciphertexts between slots — the new vault_mismatch check only *narrows* what writes it accepts. identityPub is already server-stored plaintext by spec 01 §5 (it is public key material); the lookup endpoint reveals it plus userId/displayName only to authenticated, invited household members, audited and rate-limited — no new data class, only a new authed read path over existing §5 fields. Membership (vaultId, userId, role, revokedAt) was already server-visible in the grants table; management events add audit rows containing ids and roles only, never names or decrypted content. The pubkey-substitution attack a malicious server could mount against sharing is countered client-side by the mandatory out-of-band fingerprint verification (SHA-256 of identityPub) before any VK is sealed. Keys are never persisted in plaintext on any client (identityPriv lives only in the in-memory Account, derived from the UVK-encrypted seed at unlock), UVK never rotates, and the accepted v1 residual — a removed member retaining a copied VK can decrypt ciphertext they already had — degrades future-secrecy, not the zero-knowledge property: the server itself still holds only ciphertext.

## Adversarial verification — zk_holds=false, verdict=issues

- **[blocker]** Fingerprint provenance defeats the design's sole anti-pubkey-substitution defense on native clients, breaking ZK for shared vaults against a malicious server. Kotlin Account.unlock (core/.../client/Account.kt:134-137) uses the SERVER-supplied identityPub (comment: 'pubkey comes from the server'), so identityFingerprintShort() on Android/desktop displays a hash of a server-controlled value. Failing scenario: malicious server returns P_evil as the member's identityPub both in AccountKeys (member's login) and in POST /users/lookup (owner's client); member reads fingerprint(P_evil) from native Settings, owner sees fingerprint(P_evil), OOB check passes, owner seals VK to P_evil, server opens the sealed grant and decrypts every item in the vault — and can re-seal VK to the member's real pubkey to hide the compromise. The design's zk_statement explicitly relies on this OOB verification; the web client is already safe (account.ts:111 derives from the decrypted seed) but the design's target user is the phone-first native member.
  - **Fix:** Spec 01 §5 must pin: identityFingerprint = SHA-256 of the SEED-DERIVED pubkey (boxKeypairFromSeed(identitySeed).publicKey), never the server-supplied identityPub. Kotlin Account.unlock must construct the Account with the derived pubkey (it already decrypts the seed under UVK, which the server cannot forge) and hard-fail with a security alert if the derived pubkey differs from keys.identityPub. sealOpen must also use the derived keypair. Add a vector/test asserting fingerprint(serverPub≠derivedPub) is rejected.
- **[blocker]** The vault_mismatch server belt (design §0.1 / §4.3: applyPut/applyDelete throw BadRequest) permanently wedges deployed Android 0.2.4 / desktop 0.2.0 clients mid-rollout. Because the creator's grant is deliberately UVK-wrapped, field clients CAN open shared vaults they created and will display those items; their save() bug (SyncEngine.kt:104 uses personalVaultId) then produces a put with the wrong vaultId. BadRequest fails the whole push HTTP call (AndvariApi.call throws on non-2xx), flushQueue never dequeues (dequeue happens only on per-mutation results), and every sync() runs flushQueue BEFORE pull — the doomed mutation retries forever, blocking all subsequent edits on that device until an app upgrade. Violates the stated requirement that sync keeps working for deployed clients mid-rollout. Also note BadRequest aborts the entire batch tx, rolling back innocent sibling mutations.
  - **Fix:** Make vault_mismatch a per-mutation result instead of a thrown BadRequest: return MutationResult(mutationId, "denied") (or a new terminal status old clients treat as non-retryable — 'denied' works since both client impls dequeue it), record it in the mutations idempotency table, and audit it like push_denied with meta vaultId:itemId. Old clients then drop the bad mutation, surface one error, and keep syncing; other members' data stays protected. Alternatively (or additionally) seal the owner's grant too so field clients never open shared vaults — but that contradicts the design's chosen owner-grant rationale, so the denied-result fix is required regardless.
- **[major]** Role changes never take effect client-side: both grant loops guard on !account.hasVault(grant.vaultId) (core SyncEngine.kt:44, web store.ts:60), so once a vault's VK is held, re-delivered grants (role change bumps grant rev precisely to re-deliver) are skipped and vaultRoles/roleFor never updates. Design §3.4 says 'call account.addGrant(grant)' but never removes the guard. Failing scenarios: (a) design test scenario 4 — reader promoted to writer stays read-only (UI keeps Edit hidden, reader-skip keeps suppressing conflict materialization) → SharedVaultTest scenario 4 and verify.sh go red; (b) writer demoted to reader keeps a live editor whose pushes are denied, dequeued, and the edit silently lost.
  - **Fix:** Process every incoming grant unconditionally: always update vaultRoles[grant.vaultId] = grant.role; attempt key-open (sealed or UVK-wrapped) only when the VK is not already held. State this explicitly in §3.4 (SyncEngine grants loop) and §5 (store.ts grants loop) so both impls change symmetrically.
- **[minor]** removedGrants purge does not touch the offline mutation queue. A victim with queued edits for the revoked vault hits flushQueue FIRST on next sync (SyncEngine.sync():28-31): the push is denied, dequeued, and throws ApiException 403 before pull runs — so the purge is delayed one sync cycle and the victim's offline edits vanish with only a generic error. Could also make the live-e2e removal phase flaky if B has pending writes.
  - **Fix:** In the removedGrants handler (§3.4), also drop queued mutations whose vaultId matches (add VaultCache.dropPending(vaultId) alongside deleteVault), and spell out in spec 03 §4 that a revoked member's unsynced edits to that vault are discarded (optionally exported to a local 'lost edits' note for the user).
- **[minor]** Conflict-copy materialization races across members: N clients that pull the same conflict-flagged item before any clear lands will each create a '(conflict …)' copy (SyncEngine.kt:67-81 / store.ts:100-118 have no dedupe), so a shared vault with several active writers multiplies duplicate conflict copies. Pre-existing single-user trait, but sharing amplifies it and the design is silent.
  - **Fix:** Accept-and-document in the spec 03 §5 note (household scale, harmless duplicates), or cheaply mitigate: derive the copy's itemId deterministically from (itemId, conflictedRev) — e.g. UUIDv5-style hash — so concurrent materializers converge on one copy id and the server's idempotent existing-item path absorbs the race.
- **[minor]** POST /api/v1/vaults validates only the vaultId; metaBlob and wrappedVk sizes are unbounded and any member can create unlimited shared vaults — a storage/DoS hole the item path doesn't have (items go through push quotas).
  - **Fix:** Validate metaBlob and wrappedVk as base64url with size caps (e.g. ≤4 KiB metaBlob, ≤1 KiB wrappedVk, matching the sealedVk ≤1 KiB rule) and rate-limit vault creation (e.g. limiter 'vault_create:{userId}' 5/hour) in §4.3/§4.4.
- **[minor]** The six new membership/lookup routes are reachable via the public origin. The break-glass posture (spec 03 §8) blocks register and refresh publicly and rate-limits login harder, but the design lets an attacker with one stolen access token run user_lookup enumeration and membership management from the WAN-facing origin.
  - **Fix:** Gate the membership-management and lookup routes with the same isPublicOrigin(config) → Forbidden check used by register (App.kt:229), or at minimum lookup + member add/remove. Sharing administration is a sit-at-home operation; nothing in v1 needs it over break-glass.

### 1.1 `spec/02-vault-format.md` — new §3.1 (insert after the forward-compat paragraph at :78-80, before `## 4. Vaults` :82)

New normative subsection **"3.1 login.uris — semantics & client match rules"**:
- `uris` entries are either web URIs (scheme optional, default `https`) or the native-app convention **`androidapp://<packageName>`** (Bitwarden-compatible convention).
- Normalization: lowercase host; strip a single leading `www.`; strip trailing dot; path, query, fragment, and **port are ignored for matching** (Android's `ViewNode.getWebDomain()` exposes only the domain — document the homelab caveat: two services on one host:port pair are indistinguishable, so give services distinct hostnames, which the tailnet-serve setup already does).
- Match rules (normative, all clients): a page host matches a saved host iff **exact equality** OR the page host ends with `"." + savedHost` (**label-boundary subdomain suffix** — saved `example.com` matches `login.example.com`; `evil-example.com` and `example.com.evil.net` do NOT). IP-literal hosts match exact-only. `androidapp://` matches by exact package string. **Cross-registrable-domain fill is impossible by construction** (no PSL needed in v1; PSL/eTLD+1 base-domain matching noted as a future loosening, not v1).
- webDomain trust: clients MUST honor a request's web domain only from an allowlisted trusted-browser package; otherwise fall back to package matching. ### 4.10 webDomain trust — research result (why the browser gate is required)
Any app populates its own `ViewStructure` in `onProvideAutofillStructure`, so `ViewNode.getWebDomain()` is **attacker-controlled for non-browser packages** — a malicious app can claim `webDomain=github.com` and harvest the fill. The official AutofillService "Web security" guidance: use `getWebDomain()`, canonicalize via the Public Suffix List, then verify the requesting package against the domain via **Digital Asset Links** (package name + cert fingerprint vs Package Manager) — *or* "whitelist well-known browser apps and skip the verifications, as long as the service can verify the browser"; if association can't be verified, warn/deny. v1 takes the allowlist branch (DAL requires fetching `assetlinks.json` per target domain — a network call that leaks browsing activity and adds latency; deferred to v2 for the native-app-with-web-creds case). Non-trusted packages presenting a webDomain get package-name matching only. Sources: [AutofillService API reference](https://developer.android.com/reference/android/service/autofill/AutofillService) (Web security section), [Build autofill services](https://developer.android.com/identity/autofill/autofill-services), [Digital Asset Links](https://developers.google.com/digital-asset-links/reference/rest/v1/assetlinks/check).

- **No SaveInfo / save flow** (fill-only; onSaveRequest inert).

- **No webview-specific heuristics, no ML field classification, no multi-page (username→password) session state** — a password-only page still gets password-only datasets, which covers the common two-step flows.
- **No autofill settings UI** in-app (service settingsActivity just opens MainActivity).

## ZK statement

Nothing in this feature touches the server or the wire. Autofill matching, field classification, dataset construction, and the unlock flow all run on-device against items decrypted in process memory by the existing client crypto (Account/SyncEngine in :core); the only network traffic is the pre-existing sync/accountKeys calls, unchanged in shape. No new JSON fields are sent or stored server-side — matching uses the existing `login.uris` inside the item ciphertext blob — so the server continues to store/see exactly the spec 02 §5 plaintext table (ids, revs, timestamps, sizes, kdf/auth material, ciphertext blobs) and nothing more. Keys and decrypted docs are never persisted by the autofill path: VaultSession is memory-only and cleared on lock, the unlock activity discards the password immediately after UVK unwrap, the at-rest device cache (parallel design) holds ciphertext only, and the service never logs or transmits field values, item names, URIs, or fill-request contents. The one new information flow — a requesting app's packageName/webDomain — is consumed locally and never leaves the device.

## Adversarial verification — zk_holds=true, verdict=issues

- **[blocker]** AutofillUnlockActivity constructs `AndvariApi(store.baseUrl, tokens)` with no onTokens persistence callback (and the real constructor also needs an HttpClient engine — AndvariApi.kt:48-53). Refresh tokens are single-use rotating, and reuse REVOKES THE WHOLE DEVICE (Service.kt:230-236 and the guarded-CAS at :246-254). Failing scenario on the live CT122 account: field device's access token expires → user triggers autofill while locked → unlock activity's api hits 401 on accountKeys(), tryRefresh() rotates the token pair IN MEMORY ONLY; SharedPreferences still holds the consumed refresh token. Next main-app unlock() builds a fresh api from store.load().tokens() → refresh with the consumed token → 'refresh_reuse' → all sessions for the device revoked, audit theft-signal fired, user force-logged-out mid-rollout. This will bite exactly the fielded Android 0.2.4→0.3.0 upgrade population because access-token TTLs make the 401 path routine.
  - **Fix:** Specify the construction as `AndvariApi(store.baseUrl, HttpClient(OkHttp), session.tokens()) { store.updateTokens(it) }` (identical to AndvariViewModel.newApi), and add a design invariant: exactly one token-holding AndvariApi exists at a time (the one inside VaultSession) and every api that carries tokens MUST persist rotations via SessionStore.updateTokens. Also have unlock paths reuse VaultSession's existing api when one is already bound instead of minting a second token-holder.
- **[major]** VaultSession shares one Account/SyncEngine/VaultCache between the ViewModel (viewModelScope, main dispatcher) and the autofill service (CoroutineScope(Dispatchers.Default)) with zero synchronization. InMemoryVaultCache is plain LinkedHashMaps (VaultCache.kt:38-53) and Account.vaultKeys is a plain MutableMap (Account.kt:61) mutated by addPersonalGrant during sync (SyncEngine.kt:43-47). Failing scenario: app in foreground running refresh()/sync() (cache.upsertItem/deleteItem structural mutations) while the user autofills in another window (Fold split-screen is even in the design's own test plan) → service's engine.items() iterates the map concurrently → ConcurrentModificationException (fill crash / ANR in the autofill flow) or a torn read.
  - **Fix:** Make thread-safety a stated requirement of VaultSession: either wrap cache+Account access in a mutex/synchronized owned by VaultSession, or require the cache impl to be thread-safe (synchronizedMap/ConcurrentHashMap for InMemoryVaultCache; a serialized SQLDelight connection for the durable cache) AND make Account.vaultKeys a concurrent map. Add this to the SQLDelight-coordination open question #2 as a hard acceptance criterion.
- **[major]** StructureParser extracts a single webDomain ('first non-null ViewNode.getWebDomain()') and applies it to every field in the structure. Browser pages are multi-frame: getWebDomain() is per-node, and a page with a cross-origin iframe yields fields with DIFFERENT domains in one AssistStructure. Failing scenario: attacker page evil.net (trusted browser, so webDomain honored) embeds an iframe that renders first / contains the first non-null webDomain as login.example.com — or conversely the main frame is example.com and attacker-controlled iframe fields get classified as the password field — and the saved example.com credential is filled into a field belonging to the other origin. This defeats the whole label-boundary matching rigor via frame mixing.
  - **Fix:** Track webDomain per field (propagate the nearest ancestor frame's webDomain down to each collected AutofillId). Only build datasets over fields whose own frame domain matches the item; if the structure contains more than one distinct non-null webDomain, either restrict fill to the matching frame's fields or return onSuccess(null). Add match vectors expressing 'mixed-domain form → no fill' and put the rule in spec 02 §3.1 so the future extension inherits it.
- **[major]** The label-boundary suffix rule has no minimum-label guard: a saved URI whose host is a bare TLD or single label (e.g. 'com', 'https://co.uk', 'net' — trivially arriving via spec 06 CSV import of malformed rows, which the design says is out of scope to sanitize) makes EVERY *.com page match ('evil.com' ends with '.com'). Failing scenario: an imported junk entry with uri 'com' silently offers (and one-tap fills) its credential on every .com site visited in Chrome — a cross-registrable-domain fill, exactly what §3.1 claims is 'impossible by construction'.
  - **Fix:** In parseSavedUri/matches: apply the suffix rule only when savedHost contains at least one dot (≥2 labels); single-label hosts (intranet names, TLD junk) match exact-only. Optionally reject known pure-TLD hosts outright. Add vectors: saved 'com' vs page 'evil.com' → false; saved 'router' vs 'router' → true, vs 'a.router' → false. State the rule in spec 02 §3.1.
- **[minor]** Two verify.sh-red traps in the test wiring: (1) the design's web test path `../../spec/test-vectors/urimatch.json` from web/src/vault/ resolves to web/spec/... — the real crypto test uses THREE levels up (`../../../spec/test-vectors/`, web/src/crypto/vectors.test.ts:27), so the mirror test fails at readFileSync; (2) the design calls VectorsTest's load() a 'classpath-resource pattern' but it actually reads File(System.getProperty("andvari.vectors.dir")) wired via core/build.gradle.kts:69-76 — implementing urimatch consumption as a classpath resource will fail because spec/test-vectors is not on the test resources path.
  - **Fix:** Correct the path to `../../../spec/test-vectors/urimatch.json` (or reuse the crypto test's vectorsDir helper), and describe the Kotlin side as 'same andvari.vectors.dir system-property File load as VectorsTest' — no build.gradle change needed since the property is already set for all of :core:jvmTest.
- **[minor]** Locked-path edge cases unhandled: (a) signed-out device (SessionStore.load()==null, e.g. after refresh_reuse revocation or signOut) still gets the 'Unlock andvari' setAuthentication row, and the unlock activity then has no email/tokens — dead-end UI inside another app's fill flow; (b) the setAuthentication affordance only specifies a RemoteViews lockedPresentation — on API 30+ with an inline-only IME the auth row needs the InlinePresentation overload or the chip may not render; (c) no catch-all in onFillRequest for lock races — ViewModel.lock() closes the api underneath a mid-flight service coroutine, and an uncaught exception in an AutofillService callback path crashes the app process.
  - **Fix:** (a) if store.load()==null (or accessToken empty) return callback.onSuccess(null) instead of the auth row; (b) use setAuthentication(ids, pi, remoteViews, inlinePresentation) when inlineSuggestionsRequest != null; (c) wrap onFillRequest's body in runCatching → onSuccess(null) on any throwable, and have VaultSession hand out an immutable Unlocked snapshot so a concurrent lock() can only fail the request, never NPE it.
- **[minor]** Pre-SQLDelight fallback behavior is misleading: with InMemoryVaultCache, AutofillUnlockActivity's 'best-effort sync with failure tolerated' after a CORRECT password yields an empty cache → zero datasets returned (silent no-fill that looks like a bug) AND leaves VaultSession bound, so MainActivity.start() then jumps straight to Screen.Vault showing an empty vault — which reads as data loss to the owner. Also the trusted-browser gate is package-name-only with no signing-cert check; on a device where a squatting sideload claims a browser package name not already installed, its self-declared webDomain is honored.
  - **Fix:** Until the durable cache lands, treat sync failure in the autofill unlock as a hard failure (error in the unlock card, no VaultSession.bind); flip to tolerate-offline only when items can actually be served from the SQLDelight cache. For the browser gate, note the residual risk in spec 05 and optionally pin signing-cert digests for the (small) household browser set — cheap given the list is already curated.
- **[minor]** The design leans on 'unknown-field preservation (spec 02 :78) leaves room for a future sensitive/neverAutofill flag as an additive field', but the Kotlin client can't deliver that today: Account's Json uses ignoreUnknownKeys=true and decodes into fixed data classes (Account.kt:65, 174-178), so any unknown field is silently STRIPPED when a 0.3.0 client edits an item written by a future client carrying the flag — the mid-rollout mixed-fleet case this review is explicitly guarding. Pre-existing spec-02 forward-compat violation, but this design is the first to cite it as a load-bearing extension point.
  - **Fix:** Either drop the claim from §5, or (better) file the fix alongside this feature: decode ItemDoc to a JsonObject overlay that round-trips unknown keys (as the spec requires) before the additive flag is ever introduced; add a vector/test asserting unknown-field round-trip in both impls.
