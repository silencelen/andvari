## 2. The Top 3

### 1. Item History & Restore — "the fat-finger seatbelt"
The server has silently archived the last 10 ciphertext versions of every item on every overwrite and every delete since v1 (verified: Repo.kt archiveVersion + Service.kt archives before tombstoning), and no route or client can reach a byte of it — backlog F63 calls it "spec-promised backstop as unreachable ciphertext." v6 exposes it: one additive grant-checked `GET /items/{id}/versions` (serving tombstoned items too), client-side decryption under the VK the member already holds (the AD binds itemId+formatVersion, not rev — old envelopes open with zero crypto changes), and a web History panel with "Restore this version" as a plain put over a live item. The household moment: a spouse rotates the Netflix or bank password, fat-fingers the save, and three days later recovers last week's version from the couch — no PBS drill, no SSH into CT122. Honest cost: **M, 2 reviewed batches** (server route + core decode + vectors + spec 02 §7/03 text and a design doc; then web panel), *provided* undelete, the Trash view, the diff summarizer, and the F62 passwordHistory fold are all cut from v1 — every break confirmed those four are where the L hides (clients hard-drop tombstones on ingest, verified at store.ts:380 / SyncEngine.kt:383, so "Recently deleted" needs an unpitched enumeration route and restore-marker server semantics). What could go wrong: the queued VK lazy rotation (ROADMAP:126) will make pre-rotation versions undecryptable — the design doc must record "history resets at VK rotation" and hand that invariant to the rotation design now, or the two queued features ship mutually broken; and the UI must say "up to the last 10 versions," never "nothing is ever lost."
**First slice:** grant-checked `GET /items/{id}/versions` + web History panel with per-version reveal and restore-on-live-items only — no undelete, no trash, no diff.

### 2. Stolen-phone kill switch — self-serve My Devices & Sessions
Today the T3 (lost/stolen device) response the threat model promises is admin-only (verified: the only device list and revoke routes live under /admin, App.kt:547/570) — a spouse or teen whose phone just vanished must wake the admin. This adds two self-scoped routes delegating to the exact revocation logic already shipped (AdminService.kt:81), a web "My devices & sessions" panel rendering fields the devices table already stores in plaintext (spec 02 §5 — zero new server knowledge), and a master-password-reauth gate so an unlocked-phone thief can't turn the lever on the owner. The household moment: Brynn's phone disappears at practice; sixty seconds at the family desktop revokes it and the stolen phone's offline cache wipes on next connectivity per the shipped §8 contract. Honest cost: **M, one reviewed batch** — the pitch's S-M was optimistic: F28's installId upsert (a small migration, so *not* zero-schema) must ride along or the panel shows fifteen clone "Web" rows, and wiring `notifyRevoked` (exists, zero callers) naively would sign out the *acting* device, so either deviceId plumbing lands or near-real-time is deferred to next-poll (which is all spec 05 T3 promises anyway). What could go wrong: self-lockout footguns (must always spare the acting session), and naming collision with the just-committed QW1 "devices hub" download page — the review must keep the two Settings surfaces unmistakably distinct.
**First slice:** GET /devices + ownership-checked POST /devices/{id}/revoke with reauth, web panel with current-device marker, F28 server half folded in; drop the "Secure my account" orchestration and F54 wiring.

### 3. TOTP autofill on Android — finish the login
Autofill fills username+password and then abandons the user at the 2FA prompt for an app-switch-copy-paste race against the 30-second window — on the exact surface (the Fold) the owner is protocol-testing this week. Slice 1 adds a positive `FieldKind.OTP` to the shared classifier (the negative-hint vocabulary already exists on both sides — verified identical sets in FieldClassifier.kt:36 and urimatch.ts:91), offers a fill row only when a URI-matched item actually carries `login.totp`, and computes the code at tap time via a per-dataset-auth activity so it's never stale. ZK-perfect: zero server, zero wire, the seed already lives in item ciphertext under accepted risk R6. Honest cost: **M** — but redistributed per the break: the dataset-auth activity is a *new* activity (response-level AutofillUnlockActivity doesn't return a Dataset), the web urimatch.ts mirror + shared classify vectors are mandatory, it must land in the *same* batch as F11 (which pushes the same hint sets the opposite direction), no live code ever renders in a presentation (IME processes see those), and on-Fold verification across API 29-35 is the dominant cost. What could go wrong: bank OTP pages are hint-poor, so real-world hit rate is uncertain until Fold telemetry answers; and slice 2 (clipboard-on-credential-fill) is a regression bet on the household's most fragile working surface — it stays dead.
**First slice:** slice 1 only, generic "Fill code — Bank" labels, merged with F11 as one reconciled hint-vocabulary batch.

## 3. Recommendation: prototype **Item History & Restore** this cycle

It beats the other two right now for four reasons. (1) **It's the converged signal, not one lens's taste** — 4/5 idea-round lenses and 3/10 tournament pitches independently landed on it, and the backlog's own F63 triage already says expose-or-delete; no other candidate has that mandate. (2) **It's the best impact-per-honest-dollar** — the data is already written, pruned, and listed in the spec §5 table; the M-sized first slice touches no existing route, no schema, no client crypto, and its restore is an ordinary put, so review risk is the lowest of the three despite the biggest trust payoff ("what I save can be recovered" is the identity claim a password home is remembered for). (3) **It doesn't collide with the queued lane** — cards/save-flow, the importers, and the extension spike all crowd the autofill/classifier/editor surface where TOTP-autofill lives (F11 coordination, Fold verification cycles, the same reviewers); the kill switch is entangled with F28/F54 sequencing and the fresh devices-hub naming. History lives on untouched seams. (4) **It has a deadline the others don't**: the VK lazy-rotation design is queued and will silently orphan the archive unless history's "resets at rotation" invariant is written into that design *now* — prototyping history this cycle is what forces that decision at the cheap moment. The ROADMAP nominally slots this during-soak; the honest reconciliation is that the first slice is exactly the low-risk, wire-additive kind of feature that can be built and reviewed now in the exploration lane while the pre-migration recovery cluster (F57/F59/QW-1) keeps its priority slot — and it ships v6 with a story: *v6 is the release where nothing you save is one fat-finger from gone.* Kill switch is the strongest #2 and should be the next M after; TOTP-autofill should wait for the cards/extension work to settle the classifier surface it shares.

## 4. Ideas the tournament killed

- **Emergency access / dead-man trustee escrow (Arfi)** — fatal: opening the sealed UVK grants nothing without a brand-new account-takeover auth flow the pitch never designed, and the in-band-only veto ignores the recorded ROADMAP:179 blocker (no out-of-band channel exists); real cost is the entire cycle. Do NOT re-pitch until the veto-channel design note and the claim-consumption auth design exist.
- **Sendill burn-after-read links** — fatal: the public origin is behind Cloudflare Access with a single-identity allowlist, so the anonymous recipient can never reach the link, and `clientIp()`=127.0.0.1 makes app-layer rate limiting fiction; the feature structurally un-hardens what QW-1 hardens.
- **Undelete/Trash as a v1 deliverable** — killed for this cycle (not forever): clients hard-drop tombstones on ingest, so it needs an unpitched tombstone-enumeration route plus restore-marker server semantics; it returns as its own slice after history proves the panel and F49 decides the window.
- **TOTP-autofill slice 2 (clipboard-on-credential-fill)** — converts the working instant fill into an activity round-trip on the codebase's most fragile, four-kill-switch surface, for a convenience slice 1 mostly obsoletes.
- **Hand-rolled :core QR matrix encoder (from Gestir)** — zxing-core 3.5.3 is already in the version catalog and blessed by recovery-cli for QR rendering; "byte-identical to the vendored JS" testing is technically impossible (mask selection legitimately differs).
- **Steward Panel as a new Admin card** — duplicates the existing status card for an audience of one steward; the value ships as Grafana gauges + badges in the existing Admin tables, and drill tracking belongs to the queued F59 batch, not a second parallel system.
- **F62 passwordHistory "fold-in"** — two full implementations (TS + KMP) across three editors for a field item_versions makes largely redundant; the honest SB-3 answer once history ships is probably "mark the field reserved," an owner decision, not a rider.
- **Secret-health dashboard (owner example)** — not killed, *already shipped*: web Health.tsx with strength + reuse + HIBP k-anonymity + TOTP column; strike it from future pitch lists.
