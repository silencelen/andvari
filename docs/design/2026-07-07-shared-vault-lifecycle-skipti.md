# Skipti v2 — andvari shared-vault lifecycle (delete / restore / leave / transfer / rename)

**Status:** final synthesized design (judge output of the 4-way tournament + 12 break reports). Implements F13/F14/F15 + the F19 rider; partially addresses F16 (lifecycle ops), fixes F21/F23-adjacent gaps en route. Target release: server v4 schema + client 0.5.0. All code claims below were re-verified against the repo at HEAD (file:line cited where load-bearing).

---

## 1. Overview & philosophy

The server is a courier, never an authority. It already fully controls delivery (it can withhold any vault — accepted T1), so lifecycle ops must add **no new server power** and **no new ways for a lying server to trick clients into destructive or trust-granting local actions**.

Four principles, each earned by a verified failure in a competing design:

1. **Everything tears down through the one rail every fielded client already implements** — grant revocation (`removedGrants` → dropVault/dropPending/removeVault, verified web `store.ts:194-202`, native `SyncEngine.kt:116-120`) and re-delivers through the one rail that already works — grant/vault rev-bump backfill (`Repo.pull` `v.rev>? OR g.rev>?`). Zero required changes to fielded 0.4.0 clients or the 0.2.x MSI; no minVersion bump anywhere.
2. **Never hard-delete a row that carries a sync or safety meaning.** Vault rows, grant rows, and item rows are retained forever (item rows as ciphertext-free skeletal tombstones after purge). This single rule closes the item-resurrection hole (`Service.kt:337` nulls `existing` for a missing row, skipping the `vault_mismatch` fence at :342 → fielded clients re-INSERT "destroyed" items into personal vaults) and the MSI poison-wedge (`validateAttachmentRefs` throws at :413 when rows are swept), keeps `removedGrants` correct for arbitrarily stale cursors, and means **the dormant 410/oldestRetainedRev machinery stays untouched** — eliminating the entire janitor-rev-regression break class (`currentRev = MAX(changes.rev)`, `Repo.kt:149`, is never allowed to move backward because `changes` is never pruned in v1).
3. **Destructive server ops are member-verifiable** with material the server never holds: each op carries `proof = HMAC-SHA-256(lifecycleKey, domainString)` where `lifecycleKey = HKDF-SHA-256(ikm=VK, info="andvari/v1|lifecycle", 32B)` (HKDF exists in `core/crypto/Hkdf.kt`; no new primitives). Proofs are **bound to operation identity, expiry, and sequence** (the v1 Skipti replay holes are closed, §4). 0.5.0 clients verify before trusting; verification failures degrade to *retain-and-warn*, never to purge.
4. **Mistakes are survivable by construction:** 7-day server-side grace before any destruction; a client-side **holding area** that soft-retains removed vaults + parked queued mutations until the grace ends (so a restore genuinely brings back members' in-flight edits — closing the shared "restore is lossless" lie); type-the-name confirm; a real one-click Restore.

**Explicit scope cut (judge's call):** the janitor v1 does **vault purge + transfer-offer expiry only**. No item-tombstone GC, no `changes` pruning, no mutation-dedup trim, no `oldestRetainedRev` advancement. F49's general GC stays dormant; its activation preconditions are now written into spec 02 §7 so a future activation can't repeat the three serious breaks it caused in this tournament. Household scale (≤10 users) has years of headroom in these tables.

## 2. Capabilities table

|---|---|---|---|

| Restore (within grace) | The deleting owner (resolved via their **revoked** owner grant) | must carry the `deleteId` it undoes | `POST /api/v1/vaults/{id}/restore` |

| Leave (self-removal) | Any **non-owner** active member | owner → `owner_must_transfer_or_delete` | `POST /api/v1/vaults/{id}/leave` |
| Transfer offer / cancel | Owner; target = existing **active** member with an **active** grant, ≠ self. Target may cancel (=decline). | proof (binds toUserId+offerId+expiresAt+seq), 14d expiry, one pending max | `POST /api/v1/vaults/{id}/transfer`, `DELETE .../transfer` |
| Transfer accept | The designated `pendingOwnerId` only, on a 0.5.0+ client | proof verified client-side **before** the banner renders; explicit consent; client round-trip-verifies its minted wrappedVk; acceptProof binds sha256(wrappedVk) | `POST /api/v1/vaults/{id}/transfer/accept` |
| Rename | Owner only; shared vaults only in v1 | not deleted; optional `baseVaultRev` stale-write guard; ≤4 KiB b64 | `PUT /api/v1/vaults/{id}/meta` |
| Remove member (existing route) | Owner (unchanged) | **new:** clears pending transfer if target is the pending owner | existing `DELETE /vaults/{id}/members/{userId}` |
| Admin vault override | **None — deliberately.** | Orphaned/disabled/lost owner (F14) → succession runbook: spec 04 §4 escrow recovery of the owner account (verified `applyRecovery` resets `status='active'` and touches only verifier/wrappedUvk/kdf — grants intact, `AdminService.kt:60-64`) → sign in as owner → transfer or delete → re-secure. A server route that reassigns owner rows *is* the F16 forgery class; the recovery key already outranks everything (accepted R3). | runbook `ops/runbooks/vault-succession.md` |

All lifecycle routes: authenticated + `enforceVersion` + **refused on the public break-glass origin** (and the members-GET F23 gap is patched in the same pass, with a regression test iterating every §10/§11 route) + audited ids-only. Rate buckets: `vault_destructive:{userId}` 10/h (delete, offer, rename); `vault_recovery:{userId}` 30/h (restore, cancel, accept, leave) — **restore is never blocked by the delete spree it undoes**.

Lifecycle ops are foreground REST calls and are **never queued durably offline** (prevents the delayed-retry-executes-later class).

## 3. Lifecycle proofs (the trust layer)

- `lifecycleKey = HKDF-SHA-256(ikm = VK, info = "andvari/v1|lifecycle", len = 32)` — domain-separated from the AEAD key; the server accumulates MAC pairs only under a key that never touches ciphertext.
- **Delete proof:** `HMAC(lifecycleKey, "andvari/v1|lifecycle|delete|{vaultId}|{deleteId}")` — `deleteId` is a fresh client UUID.
- **Restore proof:** `HMAC(lifecycleKey, "andvari/v1|lifecycle|restore|{vaultId}|{deleteId}")` — stored on the vault row; **consumes** the deleteId: any later tombstone bearing a consumed deleteId is stale/forged to a 0.5.0 client.
- **Offer proof:** `HMAC(lifecycleKey, "andvari/v1|lifecycle|transfer|{vaultId}|{offerId}|{toUserId}|{expiresAt}|{seq}")` where `seq = vaults.transferSeq + 1`. Accepting client rejects when local clock > expiresAt or seq ≤ its persisted last-verified seq for the vault → **canceled/expired offers cannot be replayed to completion**.
- **Accept proof:** `HMAC(lifecycleKey, "andvari/v1|lifecycle|transfer-accept|{vaultId}|{offerId}|{newOwnerUserId}|{seq}|{sha256(wrappedVk)}")` — binds the exact posted key blob, so a server-side swap of the stored wrappedVk is detectable by every keyholder; seq chaining means a replayed historical acceptProof cannot mask later ownership flips for any client that verified a newer one. Server increments `transferSeq` on accept.
- **Removal proof (grafted — closes the "guard only protects owners" break):** the existing remove-member route gains an **additive optional** body field `proof = HMAC(lifecycleKey, "andvari/v1|lifecycle|remove|{vaultId}|{targetUserId}|{nonce}")`, minted by the removing owner's (always-unlocked) client and delivered to the victim via `removedGrantsInfo`. 0.5.0 victims verify before hard-treating the removal; absent/invalid → retain-and-warn.
- **Verification is tri-state (normative):** *valid* → act + attributed notice ("deleted by its owner" is **earned by the proof** — unverified events use neutral wording, closing the manufactured-blame break); *invalid, or bare revocation of a vault whose VK we hold* → holding area + calm warning ("couldn't be verified as a real owner action — a copy is kept on this device; if nobody deleted this, tell your admin"); *unverifiable (no VK held / vault unknown locally)* → **silent no-op** (kills ghost banners on fresh devices and F20-hidden grants).
- Test vectors for every proof domain ship in `spec/test-vectors/` (web + JVM implementations must not drift).
- **Honest limits (spec 05):** proofs are MACs, not signatures — any current VK holder could mint one; server-side authz (owner principal) remains the real gate; proofs remove only *the server* from the forgery set. Full sender-signed grants (Ed25519 identity) remain deferred to P6 with rationale (new signing identity touches enrollment/escrow/every grant path). A malicious server can still *withhold* — availability-class, accepted T1.

## 4. Per-op protocol

### DELETE (soft) → grace → purge
1. **Client (owner, unlocked):** delete dialog (§9) — offers "Move items to my Personal vault first…" (F19 rider, §8) and "Back up first". Type-the-exact-vault-name gate (client-side; the name is ciphertext — verified AD `andvari/v1|vaultmeta|{vaultId}`). Mint `deleteId` + delete proof → `POST /vaults/{id}/delete {deleteId, proof}`.
2. **Server, one tx:**
   a. Resolve the caller's grant row **including revoked** (no row at all → uniform `403 not_vault_owner` regardless of vault existence — no tombstone existence oracle; grafted from D1's break).
   b. If `deletedAt` set and caller's (possibly revoked) grant has role=owner: if request `deleteId == vaults.deleteId` → **idempotent 200 {rev, purgeAt}** (replay of the committed op). If it differs (a *stale retry racing a restore*): → `409 vault_state_changed`, never a fresh delete — **operation-identity idempotency**, closing the delayed-retry-after-restore break.
   c. Else `requireOwnerOfShared`; vault must be `type='shared'`.
   d. **Capture `memberIds` BEFORE any revocation** (verified `vaultMemberIds` filters `revokedAt IS NULL`, `Service.kt:420` — computing after yields an empty WS notify set).
   e. Clear any pending transfer (audit `vault_transfer_cancel` meta `superseded`).
   f. For every **active** grant: `revokedAt=now, revokedReason='vault_delete', rev=nextRev('grant')` — per-member rev bumps are what make every member's pull emit `removedGrants` at any cursor age. Key wraps are **kept during grace** (restore must re-open them).
   g. `vaults.deletedAt=now, purgeAt=now+GRACE (7d), deletedBy=caller, deleteId, deleteProof; restoreProof=NULL; rev=nextRev('vault')`. Audit `vault_delete` meta `{vaultId}:members=N`.
3. **Post-commit:** `onChange(capturedMemberIds, rev)` → WS bell.
4. **Every member's next pull:** `removedGrants:[vaultId]` (existing, unchanged) + additive `removedGrantsInfo` entry `{vaultId, reason:'deleted', deletedAt, purgeAt, deleteId, deleteProof}`. 0.4.0 clients: today's silent purge. 0.5.0 clients: verify proof → **soft-hide into the holding area** (§6) + attributed notice; never hard-purge before purgeAt even on a valid proof (bounds tombstone-replay damage to a self-healing ≤7d soft-hide).
5. **During grace:** all reads/writes hit existing denied/403 paths (grants revoked) — zero special-casing. Pushes against the vault audit `push_denied` with meta prefix **`deleted:`** (one indexed deletedAt lookup) so intrusion review excludes routine lifecycle fallout (grafted).
6. **Purge (janitor, daily 04:30 + delayed on-boot, log-only dry-run its first armed week):** for `deletedAt NOT NULL AND purgeAt<=now AND purgedAt IS NULL`, per-vault tx that **re-checks `deletedAt/purgeAt` inside the destructive tx** (`AND` clauses on every DELETE; closes the purge-vs-restore race): delete `item_versions` (join via items), delete attachment rows (collect file paths), **convert item rows to skeletal tombstones** — `UPDATE items SET deleted=1, conflict=0, blob=NULL, blobSize=0, attachmentIds='[]'` (NO rev bump — revoked members receive nothing; rows are retained forever so `existing != null` keeps the `vault_mismatch` / edit-over-tombstone fences alive for every stale client, incl. the 0.2.x MSI, permanently) — set `vaults.metaBlob=''`, and blank **ALL** grants' key material for the vault (active + previously-revoked): `SET wrappedVk='', sealedVk=NULL` (**`''` not NULL — `wrappedVk` is `NOT NULL`, Db.kt:139**; blanket UPDATE closes the "previously-removed member's sealedVk survives" break). Stamp `purgedAt=now`; audit `vault_purge` (userId=null). Unlink blob files **post-commit, outside the DB lock**, in small batches (the single-connection ReentrantLock serializes all requests); the existing hourly orphan sweep is the crash backstop. Grafana/Loki alert: any vault with `purgeAt < now-2d AND purgedAt IS NULL`.

**Retained forever:** vault tombstone row (ids/times/proofs; metaBlob blanked at purge), all grant rows (revoked, key-less after purge), skeletal item rows, audit rows. `vaultId` is never recycled (`vault_id_taken` via `vaultType()` keeps refusing reuse — AD binding + mutation dedup depend on it). **`oldestRetainedRev` stays 0; the 410 path stays dormant.**

### RESTORE (within grace)
1. Owner: `GET /vaults/deleted` → `[{vaultId, metaBlob, wrappedVk, deletedAt, purgeAt, deleteId}]` — caller's own in-grace vaults, resolved via their revoked owner grant; ciphertext they already owned (ZK-clean). Client re-opens VK under UVK to display the name + deadline.
2. `POST /vaults/{id}/restore {deleteId}` (+ restoreProof minted client-side). Server tx: caller has an owner grant row (ignoring revokedAt) ∧ `deletedAt NOT NULL` ∧ `purgedAt IS NULL` (else `vault_gone`) ∧ request `deleteId == vaults.deleteId` (else `vault_state_changed`). If `deletedAt IS NULL` and `restoreProof` matches this deleteId → idempotent 200. Un-revoke **only** grants `WHERE revokedReason='vault_delete' AND revokedAt=deletedAt` (members removed before the delete stay removed — dedicated test), rev-bump each; clear `deletedAt/purgeAt/deleteId/deleteProof`, store `restoreProof`, bump vault rev; audit `vault_restore`; bell (notify set = the just-un-revoked members).
3. All clients re-receive vault + grants + **all items** via the existing backfill (`v.rev>since OR g.rev>since` — the proven member-resurrection path). 0.5.0 clients holding the vault in the holding area: restore local state in place, **replay parked mutations** (items were untouched during grace, so baseRevs are valid; conflicts use existing machinery), show "'Family' was restored", and durably mark the deleteId consumed. No F16 "new vault" warning fires (the holding-area record keys recognition — closes the restore-looks-like-injection break).

### LEAVE (self-removal)
1. Member confirm → `POST /vaults/{id}/leave` (no body; not durably queued).
2. Server tx: grant must exist (none → uniform 403 `no_grant`); role='owner' → `400 owner_must_transfer_or_delete`; already revoked with reason `member_leave` → idempotent 200. Else `revokedAt=now, revokedReason='member_leave', rev=nextRev`; **if caller == pendingOwnerId → clear the pending offer** (audit cancel meta `target_left`); audit `vault_member_leave`; bell (pre-captured remaining members + self).
3. Leaver's client syncs immediately → its own `removedGrants` drives the standard path; `removedGrantsInfo.reason='left'` → other own devices show the calm "You left" notice (suppressed on the initiating device). Semantics are byte-identical to owner removal (same SQL shape as `removeVaultMember`, `Service.kt:528`) — purge parity by construction; keys kept so owner re-add resurrects. Client never auto-retries leave beyond the current session (documents the re-add-then-stale-retry edge away).

### TRANSFER (two-phase offer / accept)
1. **Offer:** owner picks an existing active member → mint `offerId`, `expiresAt=now+14d`, `seq=transferSeq+1`, offer proof → `POST /vaults/{id}/transfer {toUserId, offerId, expiresAt, proof}`. Server tx: `requireOwnerOfShared` ∧ not deleted ∧ target holds an **active** grant ∧ `users.status='active'` ∧ target≠caller ∧ no pending (or overwrite-with-cancel-audit); store `pendingOwnerId/pendingOfferId/pendingOfferProof/pendingOfferExpiresAt/pendingOfferSetAt`; **vault rev bump** → the additive `WireVault.pendingTransfer` object re-delivers to every member (placing pending state on the vault row means any later cancel/expiry/re-designate clears every stale banner via ordinary row re-delivery — closes the displaced-target stale-banner break). Audit `vault_transfer_offer`.
2. **Pending is inert:** vault fully usable; owner keeps all powers; owner cancels / target declines via `DELETE /vaults/{id}/transfer`; janitor expires at `expiresAt` (audit `vault_transfer_expire`, vault rev bump). **Cleared automatically when the target leaves, is removed, or is role-changed** (belt: accept refuses when the caller's grant `addedAt/rev` postdates `pendingOfferSetAt` — closes the remove→re-add→forgotten-banner seizure break). Delete clears it. Old-client targets simply never see it (unknown field); the members panel tells the owner the target needs an updated app.
3. **Accept:** target's 0.5.0 client sees `pendingTransfer.toUserId==me` → verifies the offer proof under its held VK (embedded toUserId stops offer redirection; expiresAt + seq stop replay) → **only then** renders the consent screen (owner name + identity fingerprint + "confirm with them in person" OOB prompt — labeled advisory, not proof of sender) → mints `wrappedVk = Envelope(own UVK, VK, Ad.vk(vaultId, me))` (identical construction to vault creation) → **round-trip verifies it opens** (decrypt-and-compare before POSTing — closes the garbage-wrap path) → `POST /vaults/{id}/transfer/accept {offerId, wrappedVk, proof}` with acceptProof over `sha256(wrappedVk)`.
4. **Server tx:** `pendingOwnerId==caller` ∧ `offerId` matches ∧ not expired ∧ not deleted ∧ caller grant active ∧ grant predates the offer (2's belt). Old owner grant → `role='writer'` (their wrappedVk retained — R7-consistent, access continuous), rev bump. New owner grant → `role='owner'`, `wrappedVk=posted`; **`sealedVk` is RETAINED as fallback key material** (spec 01's exactly-one-of rule is relaxed, §7 — closes the keyless-owner F14-reborn break; both carriers are wiped together at vault purge). `transferSeq=seq`; store acceptProof + lastTransfer fields; clear pending; vault rev + both grant revs bump; audit `vault_transfer_accept` meta `{vaultId}:{from}:{to}`; bell all.
   - **Idempotent retry:** caller already `role='owner'` ∧ `lastTransferOfferId==offerId` → 200 (a lost-response retry never shows failure-after-success).
5. All clients apply the role flip via the existing unconditional `addGrant` role update (`Account.kt:260-262` verified); other members' 0.5.0 clients verify the delivered acceptProof (seq-chained) and show "'Family' is now owned by Sarah" — or a warning if invalid. If the new owner's wrap ever fails to open on a cold start, the retained sealedVk opens instead and the client **self-heals** by re-posting a fresh wrappedVk via accept-idempotent (owner-of-record re-wrap allowed with `offerId=lastTransferOfferId`).
6. Never-accepting/offline transferee: nothing changes until accept; expiry is automatic; succession for a truly lost owner remains the recovery runbook.

### RENAME
1. Owner client: `buildRenameVault` decrypts current metaBlob, updates ONLY `name`, **preserves unknown fields**, increments an embedded plaintext counter `metaV` (monotonic; clients warn-and-keep-newer when a delivered metaBlob's counter regresses — bounds the metaBlob replay corpus the rename history creates; spec honesty note added), re-seals under VK with the SAME AD → `PUT /vaults/{id}/meta {metaBlob, baseVaultRev?}`.
2. Server tx: owner grant required ∧ not deleted ∧ `requireB64 ≤4096`; if `baseVaultRev` present and `< vaults.rev` → `409 stale_meta` (client re-syncs and retries — closes silent last-write-wins over a stale tab); UPDATE metaBlob, vault rev bump; audit `vault_rename` (ids only — the name is ciphertext, verified). Delivery = the existing `v.rev>since` row path; every fielded client upserts + re-decrypts at render with zero code change. Personal-vault rename deferred (unverified metaBlob presence at enrollment).

### Concurrency matrix (all races serialize on the single-writer sqlite lock; each pair is a named test)
push-vs-delete (before: lands, grace-retains; after: per-mutation `denied`, dequeued — never a thrown 4xx); attachment-upload-vs-delete (**new:** `AttachmentStore.store` re-checks `grantRole` inside its commit tx → `no_grant`; closes the post-revocation blob-commit race for member-removal too); add-member-vs-delete (either order safe: delete re-reads active grants in-tx / add fails requireOwnerOfShared); delete-vs-offer (delete cancels pending; offer-after-delete → `vault_deleted` to grant-holders, uniform 403 to others); accept-vs-cancel/expire/delete (accept re-checks in-tx → `transfer_not_pending`); double-delete / delete-retry (idempotent by deleteId); restore-vs-purge (in-tx purgeAt re-check); restore-retry (idempotent by deleteId); leave-retry (idempotent by reason); rename-vs-delete (owner grant revoked → 403); invite-in-flight (N/A — member-add is unilateral; no vault invites exist).

## 6. Client changes (0.5.0; old clients require ZERO changes)

- **Holding area ("Recently removed", native durable / web session-scoped):** on `removedGrants` for a vault with locally-held state, 0.5.0 clients move the vault's rows + grant blob + **parked queued mutations** into a hidden holding state instead of hard-dropping (the grant blob MUST be retained or held ciphertext is dead bytes — normative). Expunge: at `purgeAt` for verified deletes; +30d (size-capped, oldest-first) for removals/unverified. Ciphertext-only at rest (spec 02 §8 invariants preserved). A live grant re-arriving (restore / re-add) reinstates state in place and replays parked mutations. Old clients keep today's hard-drop — status quo, not a regression.
- **F21 fix:** `flushQueue` returns `deniedCount` instead of throwing (`SyncEngine.kt:284`); `sync()` = flush → pull → then surface, so the purge/notice and the "N unsynced edits were parked/discarded" line land in ONE cycle. Denied mutations whose vault has an in-grace tombstone are **parked**, not dropped.
- **Save guard:** `save()` errors when `itemId` is non-null but absent from cache instead of falling back to the personal vault (`SyncEngine.kt:201`) — belt on the teleport hole; the server-side skeletal fence is the suspenders.
- **Notices (F21's "you were removed" moment):** keyed on `removedGrantsInfo` + pre-purge cached name; fire **only** for vaults present locally before the pull (no ghost banners on fresh devices / since=0 pulls — normative). Attribution ("by its owner") only when the proof verified.
- **Web:** delete/restore/leave/transfer/rename UI + Recently deleted + friendlyError mappings for every new code (F48 rider) + **a 5-minute `setInterval` sync fallback + manual refresh affordance (F29 fix rides this batch** — lifecycle makes vault disappearance routine; a WS-blocked tab must not edit a dead vault for a week).
- **Android:** notices, leave, accept-transfer consent screen, holding area. Desktop: rides `:core`; UI deferred to the owner-gated MSI refresh (F33-F36 batch).

## 7. Spec deltas (quotable)

**spec/01-key-hierarchy.md §6 (Vault keys), replacing the line at 01:101-102:**
> A grant normally carries exactly one of `wrappedVk` / `sealedVk`; after an ownership transfer the new owner's grant MAY carry both (`wrappedVk` current, `sealedVk` retained as fallback key material until vault purge). Roles: `owner` (exactly one per vault at every instant; reassignable ONLY via the two-phase offer/accept transfer of spec 03 §11, in which the new owner's client re-wraps VK under its own UVK and the old owner demotes to writer, retaining VK — R7 applies), `writer`, `reader`. Non-owner members may self-revoke (leave); the owner may not leave — only transfer or delete. Vault deletion is revocation of every grant plus server-side destruction of the vault's ciphertext after a grace window (spec 02 §7); it is NOT cryptographic erasure toward anyone who held VK (R7). Lifecycle proofs: `lifecycleKey = HKDF-SHA-256(VK, "andvari/v1|lifecycle")`; ops carry HMAC-SHA-256 proofs under it with per-op domain strings (spec 03 §11); clients verify before treating destructive server claims as owner actions. Proofs are MACs, not signatures — they remove the server, not keyholders, from the forgery set.

**spec/02-vault-format.md §4 (Vaults):**
> Vault rows gain lifecycle state: `deletedAt/purgeAt/purgedAt/deletedBy/deleteId/deleteProof/restoreProof` (soft-delete + grace + purge; vault rows are NEVER hard-deleted — `vaultId` is never recycled, normative) and `transferSeq/pendingOwnerId/pendingOfferId/pendingOfferProof/pendingOfferExpiresAt/pendingOfferSetAt`. Grants gain `revokedReason ∈ member_remove|member_leave|vault_delete` (NULL, pre-v4, reads as member_remove); restore resurrects only `vault_delete` revocations made at the matching `deletedAt`. `metaBlob` is owner-rewritable (rename) under the same VK/AD; its plaintext carries a monotonic `metaV` counter; AD deliberately excludes rev.

**spec/02-vault-format.md §5 (server-knowledge table):** vaults row += the lifecycle/transfer columns above (ids, epoch times, and opaque VK-derived MACs — PRF outputs reveal nothing about VK); grants row += `revokedReason`. No new plaintext class; membership/lifecycle topology already accepted under A6/R4. The hardening gate re-audits against this table.

**spec/02-vault-format.md §7 (Tombstones, versions, conflicts):**
> Vault deletion: 7-day grace (`ANDVARI_VAULT_GRACE_DAYS`) with all rows intact and grants revoked; then the daily janitor purges — `item_versions` and attachment rows/files deleted, item rows reduced to permanent ciphertext-free skeletal tombstones (`deleted=1, blob=NULL` — retained so the vault_mismatch/edit-over-tombstone fences outlive the ciphertext), `metaBlob` blanked, and every grant's key material wiped (`wrappedVk=''` — the NOT NULL sentinel — `sealedVk=NULL`). Vault tombstone rows, grant rows, and skeletal item rows are retained indefinitely. The janitor's v1 scope is vault purge + transfer-offer expiry ONLY. The general retention machinery (item-tombstone GC, `oldestRetainedRev` advancement, `changes`/mutation-dedup pruning) remains specified but NOT active; whenever it activates it MUST (a) never delete the `changes` row carrying MAX(rev) (`currentRev` must never regress), (b) advance `oldestRetainedRev` in the same tx as, or strictly before, any rev-history removal, keyed on one shared cutoff, (c) keep mutation-dedup retention ≥ any client's possible offline replay window, and (d) pass a quiet-server integration test (delete → 90d idle → full janitor cycle → all client generations converge without 409/410 loops).

**spec/03-wire-protocol.md §4 (Sync — pull):**
> Additive `removedGrantsInfo` (companion to the unchanged `removedGrants`, which remains the sole purge trigger): `{vaultId, reason, …tombstone/proof fields}`; reason derives from the caller's own grant's `revokedReason`. Client duties (0.5.0+): verify proofs under the still-held `lifecycleKey` BEFORE destructive local action; tri-state — valid → act with attributed notice; invalid or bare revocation of a held vault → retain in the holding area + warn; unverifiable (no key / vault unknown locally) → silent no-op. Notices fire only for vaults present locally before the pull. Denied-mutation failures MUST NOT abort the pull that delivers `removedGrants` (F21); denied mutations for in-grace vaults SHOULD be parked and replayed on restore. Clients MUST ignore unknown response fields (pins the additivity contract).

**spec/03-wire-protocol.md §10 (Shared vaults):** members GET gains the public-origin refusal (closes the F23 drift) with a regression test iterating every §10/§11 route; member-remove gains the optional removal-proof body; removing or role-changing the pending transfer target cancels the pending offer; `VaultMemberSummary.status` documented; deleted-vault special-case errors (`vault_deleted` etc.) are returned only to grant-holders — outsiders get the uniform hidden-as-403.

**spec/03-wire-protocol.md NEW §11 (Vault lifecycle):** full normative section — the eight routes of §5 above with bodies, guard order (caller-grant-first), proof domain strings and verification duties, idempotency-by-operation-identity (deleteId/offerId), rate buckets (destructive 10/h; recovery 30/h — restore never shares the delete bucket), grace/expiry defaults (7d / 14d, env-tunable), the concurrency matrix of §4, audit event names, WS-bell duty with pre-captured member sets, and the admin-succession pointer: **no admin lifecycle route exists BY DESIGN** — see `ops/runbooks/vault-succession.md` (spec 04 §4 recovery → transfer/delete → re-secure).

**spec/04-escrow-recovery.md §4:**
> Escrow and vault lifecycle are orthogonal by construction: escrow seals the UVK only — no VKs, no items. `applyRecovery` rewrites verifier/wrappedUvk/kdf and sets `status='active'`, touching neither grants nor pending transfers, so a recovered account retains every membership, ownership, and pending offer; recovery therefore doubles as vault succession for a lost/disabled owner (runbook). A vault deleted during a lockout is reconciled by the recovered account's next sync like any offline device.

**spec/05-threat-model.md (T1 + Accepted risks):**
> R7, extended verbatim: "Vault deletion removes the server's copies after a 7-day grace and triggers fleet-wide client purge; it is NOT cryptographic erasure toward ex-members — anyone granted while the vault lived may retain ciphertext and the vault key they already held, and encrypted server backups (PBS/B2) retain copies until their own retention expires (~30d). The ex-owner after a transfer retains VK likewise. True cryptographic erasure requires P6 VK rotation." T1: lifecycle ops on existing vaults are member-verifiable (the server never holds VK and cannot mint proofs); the server can still withhold, delay, or serve stale state — availability-class, accepted. Replay is bounded by expiry+sequence binding and by soft-hide-until-purgeAt; residuals: a device offline since before the newest verified seq can be shown a stale-but-genuine older state; proofs are keyholder-forgeable (server authz is the gate). Grant injection for NEW vaults (F16) remains an accepted residual until P6 owner-signed grants; the 0.5.0 "you were added to <vault>" notice is transparency for honest servers only and is NOT an F16 defense (the members list and the notice's inputs are server-controlled) — copy and spec must not claim otherwise.

**docs/ROADMAP.md P6:** owner-signed grants (Ed25519 signing identity; closes F16 fully); VK lazy rotation triggers extended to post-transfer and post-restore-with-removed-members; strike the vault-lifecycle gripe when shipped.

**NEW `ops/runbooks/vault-succession.md`:** admin + recovery sheet → offline recovery-cli → `applyRecovery` (reactivates the owner account) → sign in as owner → transfer each owned shared vault (target must accept) or delete → force password change / re-disable → verify audit trail.
**NEW `ops/runbooks/andvari-db-restore.md` (grafted, closes the snapshot-rollback wedge):** the clean binary-rollback window closes at the first lifecycle op (check `SELECT count(*) FROM vaults WHERE deletedAt IS NOT NULL OR pendingOfferId IS NOT NULL`). ## 8. F19 rider — item move/copy into a vault (ships in the same release, client-only)

Included because the delete dialog's "keep some items" story requires it; **zero server/wire change**. Web 0.5.0 Detail gains "Move to vault… / Copy to vault…" (role-gated to writer/owner on the target).

Safe shape (every tournament fatal/serious on this path closed):
- **Fresh random `newItemId` per gesture** — never deterministic across gestures (kills the moveId-collision fatal and the server-computable lineage leak). A persisted per-gesture `gestureId` derives the mutationIds so a *retry of the same gesture* converges via server dedup; a new deliberate copy gets a fresh gestureId.
- **Copy leg first, as its own push; wait for `applied`/`conflict`; only then push `delete(source)` with the baseRev OF THE CONTENT ACTUALLY COPIED** (never the current cache rev). Copy `denied` → surface role error, abort, source untouched (closes the D1 batch-coupling loss). Delete `conflict` (source changed since copy) → abort + resync + "this item changed while moving — review and retry"; never blind-retry (closes the D3 stale-copy destruction). Copy-leg `conflict` on retry (crash window) → treat as success when the server blob matches what we sent, then proceed to the delete leg.
- **Attachments included, re-encrypted with FRESH fileKeys** (download → decrypt → re-encrypt → upload under new attachmentIds bound to the new (itemId, vaultId); new fileKeys ride the new item blob) — closes the R7-capability-leak into Personal and breaks the sha256(ciphertext) linkage.
- **Bulk "move all N to Personal first" inside the delete dialog runs COPY LEGS ONLY** — originals are never deleted; the vault delete itself ends access and grace-retains them (restore then genuinely returns everything; a restore after a bulk-copy notes "items you copied to Personal are still there — you may want to remove the duplicates"). Abort mid-bulk is genuinely non-destructive; progress UI; ≤200-mutation chunking (server batch cap, Service.kt:315). Move OUT of a shared vault carries the honesty line ("members may still have copies from before the move").
- Natives get move UI with the F18 vault-picker batch (separate).

## 12. Hard questions Q1–Q14 (concrete answers)

**Q1 — Fate of items on delete:** Protocol destroys after grace; the UI offers rescue first. Grace (7d): item rows, item_versions, attachment rows+files all intact and unreachable (grants revoked; pull joins active grants) — restore is lossless server-side, and 0.5.0 clients park+replay members' in-flight edits. At purge: item_versions and attachment rows/files destroyed (files unlinked post-commit outside the lock; orphan sweep backstops crashes), item rows reduced to permanent skeletal tombstones (no ciphertext; keeps the vault_mismatch/edit-over-tombstone fences alive forever), metaBlob and ALL grant key wraps blanked (`wrappedVk=''` NOT-NULL sentinel, `sealedVk=NULL`) — post-purge the server holds zero ciphertext and zero wrap material. Move-to-owner is a client-side pre-step (F19 rider, copy-legs-only, fresh itemIds and fresh attachment fileKeys) because the server cryptographically cannot re-encrypt (AD binds vaultId). Escrowed copies: none exist — escrow seals UVK only (spec 04 §3), zero interaction. Honest residue named in copy + spec: ex-member copies/capability (R7) and PBS/B2 encrypted backups until their retention (~30d) expires.

**Q2 — Who may delete; admin override (F14):** Owner only, from an unlocked client. NO admin vault route, deliberately: server-assignable ownership/deletion is exactly the F16 forgery class, and D2's admin routes grew two independent breaks (pending-survives-recovery; casual override). Orphaned vaults: succession runbook — recover the owner account via spec 04 §4 (verified: `applyRecovery` restores `status='active'` with grants intact), act as the owner, transfer or delete, re-secure. Zero new attack surface; priced under accepted R3; fully audited. A native admin route is deferred until a second non-owner admin exists.

**Q3 — Owner leave:** Blocked (`owner_must_transfer_or_delete`). Exits: transfer-then-leave, or delete. Invariant: exactly one owner at every instant — no ownerless-live vault can be created, which is what keeps F14 a recovery-time problem instead of a steady-state one.

**Q4 — Member leave vs owner removal:** Byte-identical revocation on a new self-scoped route (the existing members route keeps `cannot_target_self`): `revokedAt + rev bump`, keys kept (re-add resurrects), `revokedReason='member_leave'` vs `'member_remove'` drives only audit type and notice copy. Purge parity is the same code: the leaver's own next pull delivers removedGrants → identical dropVault/dropPending/removeVault (0.5.0: holding area) on every device; the initiating client syncs immediately and suppresses the banner. Leaving while pending-transfer-target auto-cancels the offer.

**Q5 — Transfer protocol; offline transferee; audit:** Two-phase offer/accept between existing members, because the crypto demands it — the server can never mint a UVK wrap; only the transferee (who already holds VK via sealedVk) can self-upgrade. Offer: proof-bound (toUserId+offerId+expiresAt+seq) vault-row annotation; inert; cancellable; 14d expiry; cleared on target remove/leave/role-change. Accept: client verifies the proof BEFORE showing consent, round-trip-verifies its minted wrappedVk, acceptProof binds sha256(wrappedVk)+seq; server atomically swaps roles, RETAINS the new owner's sealedVk as fallback (closing the keyless-owner trap), increments transferSeq. Offline/never-accepts: nothing changes; offer expires; lost owner → recovery runbook. Retries idempotent by offerId. Audit: `vault_transfer_offer/accept/cancel/expire` with `{vaultId}:{toUserId}` ids only; all members' 0.5.0 clients verify the seq-chained acceptProof and can detect server-side role rewrites or wrap swaps.

**Q6 — Rename (F15):** The name is E2E ciphertext today — inside `metaBlob`, AEAD under VK, AD `andvari/v1|vaultmeta|{vaultId}` (verified; server knows vaults only as ids) — and stays so. Rename = owner's client read-modify-writes the metaBlob (only `name`, unknown fields preserved, `metaV` counter incremented), `PUT /vaults/{id}/meta` with optional `baseVaultRev` (stale tab → `stale_meta`, no silent last-write-wins); server validates b64 ≤4 KiB, bumps vault rev; delivery is the existing `v.rev>since` row path — every fielded client re-decrypts at render with zero changes; the 0.2.x MSI shows no vault names (no-op). Owner-only; shared vaults only in v1; refused on deleted vaults. Replay of old metaBlobs is bounded by the `metaV` warn-and-keep-newer rule and honestly documented as a T1 stale-data power, never claimed eliminated.

**Q7 — Tombstones; retention; stale-cursor 0.4.0 proof:** Retained forever: vault tombstone row (`vaultId` never recycled — `vault_id_taken` keeps refusing; AD-binding and mutation dedup depend on it), all grant rows (the removedGrants delivery vehicle for ANY cursor age; key-wraps blanked at purge), skeletal item rows, audit. Retained 7 days: all ciphertext. GC/410: deliberately NOT activated — `oldestRetainedRev` stays 0, so the never-exercised 410 path stays theoretical; its future activation invariants are now normative in spec 02 §7. Stale-cursor trace (verified code): pull joins `revokedAt IS NULL` grants → the dead vault's rows are simply absent; `removedGrants` (`revokedAt NOT NULL AND rev>since`) hits because delete bumped every member's grant rev → client runs its shipped purge; there is no per-vault GET anywhere, so no 404 is reachable; pushes → per-mutation `denied` → dequeued before the throw (SyncEngine.kt:279 — the queue physically cannot wedge); members/attachments GETs → non-fatal 403. No wedge, no corruption, at any cursor age.

**Q8 — Offline members:** Cached ciphertext stays readable until next contact — spec 05 T3's accepted narrowing, and R7 means revocation was never capability-revocation; copy says so. Queued mutations on reconnect, traced: flush pushes → per-mutation `denied` → each dequeued → (0.4.0) throw aborts the cycle → next cycle purges (≤2 cycles, F21). 0.5.0: deniedCount deferred until after pull; mutations for in-grace vaults are PARKED in the holding area and replayed on restore — an accidental delete no longer destroys a spouse's offline edits; discarded-for-real only at grace end, with an honest notice. Convergence guaranteed: grant rows retained forever → removedGrants reaches any cursor; never-reconnecting devices keep ciphertext forever (stated in copy).

**Q9 — Concurrent writers; invite-in-flight; racing lifecycle ops:** The single-connection SQLite lock serializes everything; §4's matrix enumerates every pair with its typed outcome and a test: push-vs-delete (denied, dequeued — never thrown), attachment-upload-vs-delete (new in-tx grant re-check in AttachmentStore.store), add-vs-delete (either order safe), delete-vs-transfer (delete cancels pending; late accept → transfer_not_pending; delete-by-ex-owner → not_vault_owner), restore-vs-purge (in-tx purgeAt re-check), double ops (idempotent by operation identity — deleteId/offerId — so a delayed retry after a restore gets `vault_state_changed`, never a fresh delete). Invites-in-flight: vault invites don't exist (member-add is unilateral) — nothing to race.

**Q10 — Grant authenticity (F16):** Split verdict. FIXED NOW for lifecycle ops on existing vaults: proofs under `lifecycleKey` (HKDF of VK) mean the server — which never holds any VK — cannot mint a delete, removal, offer, or accept for a vault the family uses; forged transfer requires VK possession AND a consenting human whose banner only renders after verification; forged delete/removal degrades from silent data loss to retain-and-warn. Bindings (expiry, offerId/deleteId consumption, transferSeq chaining, sha256(wrappedVk)) close the replay holes the tournament found. DEFERRED with rationale: sender-authenticated grants for NEW vaults need a signing identity (X25519 box keys can't sign) touching enrollment/escrow/every client — P6, where VK rotation rebuilds grant plumbing anyway; VK-HMAC can't help there (the server legitimately knows a forged vault's VK). Interim transparency: added-to-vault notices, explicitly labeled NOT a defense (their inputs are server-controlled). Residuals stated plainly in spec 05: proofs are keyholder-forgeable MACs; the server retains withhold/delay powers.

**Q11 — Escrow interaction:** Structurally nil, verified at both layers: escrow seals exactly the UVK; `applyRecovery` (AdminService.kt:60-64) rewrites verifier/wrappedUvk/kdf + `status='active'` and never touches grants or pending state — a recovered account is still exactly the member/owner it was (this IS the succession primitive, Q2). A recovered member whose vault died mid-outage converges via removedGrants like any offline device. No admin-initiated pendings exist to survive recovery (no admin routes). Server DB backups retain deleted ciphertext through their retention window and an operator restore could resurrect an in-grace vault — documented in spec 05 and the restore runbook, since backups only ever hold what T1 sees.

**Q12 — Mistake-proofing:** Five layers: type-the-exact-name confirm (client-side — only clients can see names); copy-items-first and back-up-first affordances in the dialog; a REAL 7-day undo (one-click Restore from Recently deleted, rebuilding every member via proven backfill); the 0.5.0 holding area + parked mutations making even members' in-flight work survive an accidental delete-then-restore; rate limits with restore on a separate generous bucket (an undo is never blocked by the mistake it undoes). During grace the family sees the vault GONE (access truly is revoked — a half-alive ghost would lie on old clients and invite denied-edit loss); 0.5.0 members see the honest "restorable until…" notice. Restore surprises no one: holding-area recognition suppresses new-vault warnings and replays parked edits.

**Q13 — Audit + notification:** Audit rows (ids/roles/counts only — user_lookup lesson; names can't leak, they're ciphertext): `vault_delete {vaultId}:members=N`, `vault_restore`, `vault_purge` (userId=null), `vault_member_leave {vaultId}:{userId}`, `vault_transfer_offer/accept/cancel/expire {vaultId}:{toUserId}`, `vault_rename {vaultId}`, plus `push_denied` with the new `deleted:` prefix for lifecycle fallout (keeps intrusion alerting clean). All in the op's tx, shipped to Loki as today. In-app (closes F21's void): removedGrantsInfo + pre-purge cached names drive the §11 notices — deletion (attributed only when proof-verified), removal, you-left, transfer offer/completion, restore, discarded/parked-edit counts; fired only for locally-held vaults; WS bell rings every affected user from pre-captured member sets.

**Q14 — minVersion/426:** No bump, any platform — a design requirement met by construction: teardown rides removedGrants (fielded since 0.3.0; MSI converges by relaunch), denials ride per-mutation `denied` (all clients, always dequeues), and every wire change is a new route or ignored field. Arming a pin today would be actively harmful: the 426 gate is advisory (F37) and the MSI self-reports platform 'android' (F36) — an android pin meant for phones would 426 the owner's daily driver into a raw error with no upgrade UI. ## 14. Residual risks (accepted, documented)

1. The janitor's vault purge is andvari's first data-destruction code: mitigated by 7d grace, in-tx re-checks, dry-run week, CT122 snapshot, PBS/B2 (~30d) backstop, per-vault audit, and the purge-stall alert — residual accepted.
2. Proofs are MACs: any current keyholder could mint one; server-side authz is the real gate. If a future authz bug exposes a lifecycle route, proofs won't stop a malicious family member. Signatures land with P6.
3. A malicious server can still withhold vaults, delay lifecycle delivery, or serve one member stale-but-genuine state older than their last verified seq — availability/staleness class, accepted T1.
4. Old (0.4.0) clients still hard-drop on removedGrants and discard denied offline edits; the grace window cannot protect them (their dequeue behavior is fielded — verified). Bounded regression: exactly today's behavior; fixed by updating.
5. The 0.2.x MSI shows a deleted vault's stale items for the rest of its session and, post-transfer, retains legitimate writer powers including item deletes — pinned by test, solved for real by the owner-gated MSI refresh (F33-F36), after which an optional plataform-true minVersion pin becomes available.
6. `changes`/`mutations` tables keep growing (F56) — deliberate v1 trade for wedge-immunity; activation preconditions for pruning are now normative in spec 02 §7.
7. Grace means "deleted" ciphertext persists server-side 7 more days (privacy inversion) — stated in the dialog; no purge-now button in v1 (undo beats instant shredding at family scale).
8. Restore after a bulk copy-to-Personal leaves duplicates for the owner to clean up (safe-direction failure, noted in the restore flow).
