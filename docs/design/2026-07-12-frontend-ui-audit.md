# Andvari — Frontend & UI Design Audit (All Platforms)

> **Date:** 2026-07-12 · **Repo:** `dev-ledger/andvari` @ HEAD `46e1e4e` · **Auditor:** huginn (multi-agent)
> **Versions audited:** server/web 0.16.0 · extension 0.12.0 · android/desktop 0.15.0 (code 0.16.0)
> **Scope:** visual design · cross-platform consistency · accessibility · UX/flows · responsive/layout · microcopy · interaction/feedback · password-manager trust-UX.
> **Out of scope:** crypto/security correctness (separate review track), backend logic, test coverage.

## Context

Andvari is a zero-knowledge household password manager that grew fast across **five UI surfaces**, each authored independently. It had never had a dedicated design/UX audit — prior effort was security/crypto review. This audit covers every surface a household member can encounter, through eight design lenses, with live-rendered screenshots of the web app (fixture-rendered, no real secrets) and code-level analysis of the native clients (which have no headless render path here).

## Method

1. **Reconnaissance** — three parallel agents mapped every surface's screens, design tokens, theming, and a11y provisions.
2. **Live capture** — the web app was built and driven headless (puppeteer + request-interception with synthetic fixtures — no server, no real secrets) to produce **23 screenshots** across screens, states, both themes, and the mobile breakpoint. Native (Android/desktop) has no `@Preview` and is audited from source.
3. **Audit workflow** — a grid of **20 finder agents** (5 surfaces × 4 lens-clusters) plus a cross-platform consistency stage produced 150 raw findings; a **per-surface adversarial verification** pass (each finding re-opened at its cited `file:line` and challenged) confirmed **131** and rejected **9**; a synthesis stage produced the themes and prioritized backlog below.

## Executive summary

andvari has a strong, coherent brand concept and genuinely good security primitives — zero-knowledge design, clipboard auto-clear, a shown-once recovery ceremony — but the execution is applied unevenly across its five surfaces. Web is the reference implementation (it alone has an AA-tuned --link token, tofu-proof SVG brand geometry, a two-tone wordmark, and a contrast unit test); the extension, Android and desktop clients are hand-synced copies of the design system that have already drifted, and the server-side HTML/email shell was never in the accessibility pass at all. The single biggest through-line is that the design language lives in five disjoint hand-maintained files with no shared seam and a contrast test that guards only two tokens — so light-theme WCAG AA failures on security-critical text (worst: Android's primary error banner at 2.56:1, roughly half the floor), invisible focus indicators, dead tokens, and baseline-Material lavender bleed have all shipped uncaught. The brand's own tofu-proofing decision (drawing the rune as SVG) was made only on web, so the primary mark risks rendering as empty boxes at every trust-critical screen on native and the injected overlay. Trust-UX is the other soft spot: the core copy action gives no feedback and the auto-clear window is never communicated, onboarding over-states a printed recovery sheet the default emailed (waived) flow never produces, and the invite email — every member's first contact — arrives as an unbranded no-reply plaintext message indistinguishable from a phishing lure. Native lags the most and is the hardest to visually iterate, so defects that look fine in dark mode hide in light mode and in un-overridden Material container roles. Encouragingly, most of this is cheap: the majority of findings are pure color-value or copy changes, and the highest-leverage structural move is consolidating the palette into one generated token source with a broadened contrast test so these defect classes cannot regress.

## Cross-cutting themes

- **[High] Fragmented design system — five hand-synced token copies with no shared seam** — The palette is re-declared verbatim across web CSS, extension popup.css, the content-ui overlay, Android Theme.kt and desktop Main.kt with nothing mechanically coupling them; sync is comment-only and per-file contrast tests never assert cross-file equality. This is the root cause of nearly every divergence below — drift has already shipped as AA failures, dead tokens, and light-scheme gaps.
- **[High] Light-theme WCAG AA contrast failures on gold/danger/faint ink and focus indicators** — Gold-as-text (~3.7-4.05:1), danger text on tinted strips (~4.39:1), Android's error banner on errorContainer (2.56:1), and light-mode focus rings (~2:1) all fall below AA/3:1 across web, extension, Android and desktop — concentrated on the exact wrong-password/lock/error text a security tool most needs legible. The contrast tests give a false green because they only check two tokens.
- **[High] Native Material theming incomplete — baseline purple/pink bleeds into the gold treasury** — Android and desktop override only the legacy base color roles, leaving M3 container/tertiary/errorContainer roles at Material baseline, so Cards, the Add FAB, dialogs and warnings render in stock lavender/mauve and the light scheme drops surfaceVariant/onSurfaceVariant. The treasury identity is largely absent on-device — the generic Material look the brand is trying to avoid.
- **[High] Clipboard auto-clear and copy actions give no feedback; sensitive seeds never clear** — Copy — the core daily action — produces no in-app confirmation on native/desktop and the auto-wipe window is never communicated on any surface, so a late paste silently fails and a real safety feature is undiscoverable. Worse, Settings' TOTP seed/otpauth copies never auto-clear at all.
- **[High] Onboarding stakes under-communicated — recovery-sheet copy is wrong for the default flow** — The first enroll gate (and the invite email) tell every invitee they need a printed recovery sheet, but the default emailed/waived flow generates the member's own phrase on-device and produces no sheet — stalling the most important ceremony. 'Recovery' is overloaded four ways and forgotten-password paths dead-end.
- **[High] Unbranded plaintext invite email is a phishing-indistinguishable first contact** — The highest-stakes first message arrives as unstyled plaintext from a raw no-reply mailbox with no inviter name, no wordmark, no anti-phishing inoculation line, and an opaque 223-char link — a recipient has no legitimacy signal to tell it from a credential-harvest lure.
- **[Medium] Brand mark ships as raw runic codepoints (tofu risk) on every surface but web** — Web draws ᛅ/ᛝ as SVG geometry precisely because the codepoints tofu where no Runic font is installed and CSP bars web fonts; the extension, Android (Noto Serif has no Runic block) and desktop all print the raw codepoint as text, re-introducing the exact failure at loading, unlock, empty-state and the anti-spoof overlay.
- **[Medium] Screen-reader/keyboard a11y gaps in dynamic states** — Desktop has zero live regions; Android autofill errors are silent; copy buttons announce a bare 'Copy'; search fields and the recovery-confirm input lack programmatic labels; busy states aren't announced and disabling drops focus; headings/landmarks/skip links are missing. Static contrast was largely fixed but the async and control-naming layer wasn't.
- **[Low] Member-facing copy leaks ops/dev jargon** — 'break-glass', 'the public address', 'escrow', 'TOTP secret (otpauth:// URI or base32)', 'Enroll', 'Invite token', and cvv/exp/num abbreviations land on the most anxiety-prone security screens of a warm-household product, clashing with the established 'unsealing your hoard' voice.
- **[Low] Absent first-impression, platform-fit and progress polish** — No favicon/theme-color/apple-touch-icon/PWA manifest/OG meta/noscript on the web shell; no 'Unsealing…' progress affordance during the multi-second KDF; desktop MSI/Deb ship the stock Compose icon; and within-surface drift (7-15 font sizes, seven corner radii, ragged coin monograms, single-color wordmark on native) undercuts the crafted-treasury feel.

## The headline: one design language, five hand-synced copies

The "treasury/gold" system is re-declared **verbatim in five disjoint files** with **no shared seam** — kept in sync only by hand-written comments. This is the root cause of most findings below: drift has already shipped as WCAG failures, dead tokens, and light-scheme gaps, and the per-file contrast tests never assert cross-file equality.

| Token source | File |
|---|---|
| Web (de-facto origin) | `web/src/ui/styles.css` |
| Extension popup | `extension/popup.css` |
| Content-script overlay | `extension/src/content-ui.ts` (`--anv-*`) |
| Android | `app-android/.../app/Theme.kt` |
| Desktop | `app-desktop/.../desktop/Main.kt` |

**Shared dark palette** (identical hexes across the copies): gold `#d0a94a`, gold-bright `#e8c66a`, bg `#14120e`, surface `#1e1b15`, ink `#ede4d0`, ink-dim `#a79c85`, ink-faint `#8d8370`, danger `#cf6b5a`, radius 10px. **Known drift:** the overlay's `--anv-ink-faint` is still the pre-AA value (`#6f6752`/`#9a8f74`) that web+popup bumped to `#8d8370`/`#786c50`, and the overlay is covered by no contrast test.

### Run-2 executive summary

196 verified findings, zero criticals, and a clear shape: andvari's hard security architecture and its dark-mode web experience are genuinely strong — verifiers repeatedly confirmed disciplined engineering (inline-confirm idioms, error canons, live regions, contrast tests, deliberate a11y passes). But every discipline is applied island-by-island, and the dominant failure mode is drift: the fix that landed on one surface never propagated. The web's --link AA fix is missing from the extension popup; a11y passes skipped Detail rows and the autofill overlays; five hand-synced palette declarations diverged three separate ways, once inside a single commit labeled "across all four clients." The consequences cluster hard: (1) the light theme fails WCAG AA product-wide (gold text, focus rings, button labels, error colors) while dark passes; (2) both Compose clients leak Material purple/pink onto security ceremonies because ~18 color roles were never declared; (3) the core copy/fill loop gives silent or false feedback everywhere, with an undisclosed clipboard auto-wipe that reads as breakage; (4) a handful of one-tap paths destroy drafts, offline caches, or the recovery guarantee itself — including a desktop "un-skippable" recovery gate that the app's own idle lock silently skips; (5) the first-contact invite email is factually wrong on its default path, anonymous, and feedback-free. For a product whose credibility is "looks trustworthy and usable by every household member," the worst gaps sit exactly on the trust surfaces: first email, locked screens, recovery ceremonies. Do first: the token-level light-theme color sweep, the invite-email fix, and the batch of one-dialog guards — all S-effort, high-severity. Immediately after, build the cross-client token-parity + contrast harness so the color fixes can never silently regress, then work the M-effort feedback, Compose-theming, and recovery-integrity clusters.
